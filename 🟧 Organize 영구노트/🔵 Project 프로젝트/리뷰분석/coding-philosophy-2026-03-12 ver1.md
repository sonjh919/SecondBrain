## 코딩 철학 원칙

### 1. 의도를 이해한 뒤에 수용한다
피드백을 받으면 단순히 "수정하겠습니다"로 끝내지 않고, **왜 그래야 하는지**를 반드시 질문하고 이해한 후에 반영한다. 모르는 것을 모른다고 말하는 것을 두려워하지 않는다.

> **근거**: "앗 수정하겠습니다" 같은 단순 수용보다, "그렇다면 혹시 검증의 역할도 외부에서 수행한 후 생성자 파라미터로 전달해 주는 것이 더 좋다고 생각하시나요? 저는 검증 또한 비즈니스 로직의 일부라고 생각하여..." 처럼 자신의 관점을 밝히고 대안을 제시하는 패턴이 반복됨 — PR #521

### 2. 트레이드오프를 의식적으로 고민한다
설계 결정에서 양쪽의 장단점을 비교하며 고민하는 습관이 있다. 단순히 "이게 좋다"가 아니라 "A와 B 사이에서 어떤 것이 더 나은지"를 따진다.

> **근거**: "요구사항을 한 곳에 몰아넣어 유지보수를 편리하게 하는 설계 vs 컨트롤 불가능한 동작을 외부에서 주입받는 설계 두 가지 경우에서 트레이드오프를 고려해야 할 것 같은데, 어떤 경우가 더 좋다고 생각하시나요?" — PR #661

### 3. 실용성을 중시하되 원칙의 이유를 알고 싶어한다
"이건 붙이라고 했어!"식의 맹목적 따르기를 경계하며, 원칙의 실질적 효용을 확인하고 싶어한다.

> **근거**: "이론상으로는 값의 변경을 막는다는 이유로 필요하다는 것을 잘 알지만, 실제로는 지금까지 메소드 파라미터에 final을 전부 빼버려서 이슈가 발생한 적은 한번도 없어요. 그래도 '이건 붙이라고 했어!'라는 마인드로 계속 붙이는 것이 좋을까요?" — PR #521

### 4. 스스로 문제를 발견하고 해결 방향을 먼저 세운다
리뷰어가 지적하기 전에 스스로 코드의 문제를 인지하고, 해결 방향을 먼저 고민한 뒤 리뷰어에게 검증받는 방식을 취한다.

> **근거**: "오늘 조금 더 고민을 해 본 결과, 다음과 같은 해결 방법을 얻었습니다. 1. 이 로직을 전부 WinningLotto로 옮기고... 2. 이렇게 하였더니 결합도가 너무 올라가고... 3. 현재 상속 대신 합성을 이용하여 리팩토링을 진행하고 있습니다." — PR #521

### 5. 학습한 개념을 즉시 실전에 적용하려 한다
책이나 외부에서 배운 개념을 현재 코드에 바로 적용해보며 체득하는 스타일이다.

> **근거**: "모던 자바 인 액션 책에서 람다와 동작의 파라미터화 부분을 최근에 보고 있었는데, 마침 리뷰로 활용 방법을 알려주셔서 덕분에 해당 개념을 더 잘 체득할 수 있게 된 것 같습니다." — PR #661

## 코딩 스타일 패턴

### 선호하는 패턴

- **MVC 패턴 분리**: 1단계부터 Controller, Domain, View를 분리하여 구현한다. 리뷰어도 "1단계에서부터 MVC 패턴에 맞춰 잘 분리해주셨던터라 크게 피드백드릴 부분이 없었어요"라고 칭찬.
  > "2단계도 빠르게 잘 구현해주셨습니다. 1단계에서부터 MVC 패턴에 맞춰 잘 분리해주셨던터라 관련해서는 크게 피드백드릴 부분이 없었어요." — PR #661, Rok93

- **테스트 메서드 한글 네이밍**: `@DisplayName` 대신 메서드명 자체를 한글로 작성하는 것을 선호한다. 이유는 가독성 차이가 크지 않으면서 코드가 간결해지기 때문.
  > "2번과 3번의 경우에는 큰 차이점을 모르겠어요. 오히려 3번의 경우는 @DisplayName 때문에 메서드의 길이가 길어지는 단점이 있다고 생각합니다." — PR #521

- **인터페이스를 통한 테스트 용이성 확보**: 테스트하기 어려운 동작(랜덤 등)을 인터페이스로 분리하여 테스트 시 대체 가능하게 설계한다.
  > "Generator라는 인터페이스를 따로 분리한 이유는, 만약 테스트 과정에서 Lotto 객체를 랜덤으로 생성해야 할 일이 생긴다면 FakeGenerator라는 클래스를 따로 만들어 테스트용으로 사용하기 위한 목적이었습니다" — PR #521

- **정적 팩토리 메서드 활용**: 객체 생성 시 생성자보다 정적 팩토리 메서드를 선호한다. 2단계에서 `WinningLotto.of()`, `LottoNumber.from()` 등으로 리팩토링.

- **검증을 도메인 객체 안에서 수행**: 검증은 비즈니스 로직의 일부이므로 도메인 객체 내부에서 처리하는 것을 선호한다.
  > "저는 검증 또한 비즈니스 로직의 일부라고 생각하여 객체 안에서 진행하였는데" — PR #521

### 지양하는 패턴

- **무의식적 관성 코딩**: 과거 경험(프리코스, 스프링)에서의 습관이 무의식적으로 나오는 것을 경계한다. `from()` 네이밍 남용, 문자열 입력 그대로 사용 등을 스스로 인지하고 반성.
  > "프리코스 때 정적 팩토리 메서드를 사용하면서 from으로 네이밍을 했었던 경험이 무의식적으로 다시 튀어나온것 같네요." — PR #521

- **불필요한 분리**: 클래스가 2개이면 Factory도 2개 만들어야 한다는 무의식적 사고를 경계.
  > "클래스가 2개이니 당연히 팩토리도 2개를 만들어야 한다는 생각이 무의식에 있었던 것 같아요!" — PR #521

## 리뷰 스타일 패턴

> 이 PR들에서 손준형은 리뷰를 **받는** 입장이므로, 리뷰를 받아들이는 스타일을 분석한다.

### 리뷰어가 자주 지적하는 것 (= 개선 영역)

1. **네이밍 정확성** (6회) — 메서드명, 변수명, 파라미터명, DTO prefix 등 네이밍이 의도를 정확히 전달하지 못하는 경우가 반복됨.
   > "가급적 getXXX, setXXX 보다는 메서드의 행위가 드러나는 네이밍을 고민해보시면 좋을 것 같아요." — PR #521

2. **접근 제한자/final 누락** (4회) — 필드에 final 빠뜨림, 상수가 public으로 노출, IDE 자동생성 기본값을 확인하지 않음.
   > "상태 값이 한번 할당된 뒤부터는 변경되지 않는 것 같은데요. final 키워드를 사용해서 재할당을 막아보면 어떨까요?" — PR #521

3. **생성자 역할 과부하** (3회) — 생성자에서 파싱, 검증, 변환을 모두 수행. 생성자는 생성 역할만 하도록 분리 권장됨.
   > "생성자는 가급적 객체를 생성하는 역할만 수행하도록해보면 어떨까요?" — PR #521

4. **패키지 구조** (3회) — 도메인 종속 코드를 `global` 패키지에 넣는 습관. Exception, Generator, Factory 모두 도메인에 가까움.
   > "global이라는 패키지는 어떤 경우를 의미하는걸까요? 예외 메시지도 생각보다 로또 도메인에 종속된 내용들이 많아서 조금 혼란이 생기는 것 같아요." — PR #521

5. **테스트 누락** (2회) — 핵심 로직에 대한 단위 테스트가 빠지는 경우.
   > "matchCount 메서드에 대한 테스트 코드가 누락된 것 같은데, 추가 부탁드려요." — PR #661

6. **매직 넘버** (2회) — 숫자 리터럴을 상수로 추출하지 않음.
   > "1, 45, 6은 각각 어떤 숫자일까요? 👀" — PR #521

### 리뷰어가 자주 칭찬하는 것

1. **피드백 반영 속도와 품질** — 피드백을 빠르고 정확하게 반영함.
   > "링크 피드백 반영 잘해주셨습니다. 👍💯" — PR #521, PR #661 모두

2. **아키텍처 분리** — MVC 패턴 분리가 초기부터 잘 되어 있음.
   > "1단계에서부터 MVC 패턴에 맞춰 잘 분리해주셨던터라 관련해서는 크게 피드백드릴 부분이 없었어요." — PR #661

3. **치열한 고민** — 설계 결정에서 깊이 있는 사고를 보여줌.
   > "치열하게 고민 중이시군요? 😃 결과물이 벌써부터 기대되네요. 🤩" — PR #521

### 리뷰 우선순위
리뷰어의 지적 빈도와 사용자의 반응을 종합하면, 다음 가치를 순서대로 중요시하는 것으로 보인다:

1. **설계적 정합성** — 상속 vs 합성, 역할 분리, 도메인 응집 등 구조적 결정에 가장 많은 에너지를 쏟음
2. **테스트 용이성** — 테스트 가능한 설계(인터페이스 분리, 함수형 인터페이스)를 의식적으로 추구
3. **명시성과 가독성** — 한글 테스트명, distinct() 선호, 의도가 드러나는 코드 선호
4. **실용주의** — 원칙을 따르되 실질적 효용을 확인하고 싶어함 (final 논의)
