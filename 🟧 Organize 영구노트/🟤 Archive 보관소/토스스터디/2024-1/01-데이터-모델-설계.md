# 데이터 모델 설계

## 개요
1억 계정 규모의 비밀번호 초기화 기능을 지원하기 위한 데이터베이스 스키마 설계입니다.

## 주요 테이블

### 1. users 테이블
사용자의 기본 정보와 현재 비밀번호를 저장합니다.

```sql
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    current_password VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_updated_at (updated_at)
);
```

**설계 포인트:**
- `user_id`: BIGINT 타입으로 1억 이상 수용
- `current_password`: BCrypt/Argon2 해시 저장 (255자 충분)
- 인덱스 최소화로 INSERT/UPDATE 성능 확보

---

### 2. password_history 테이블
사용자의 비밀번호 사용 이력을 저장합니다.

```sql
CREATE TABLE password_history (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    used_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_user_recent (user_id, used_at DESC),
    INDEX idx_used_at (used_at),
    
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE
) PARTITION BY RANGE (YEAR(used_at) * 100 + MONTH(used_at)) (
    PARTITION p202410 VALUES LESS THAN (202411),
    PARTITION p202411 VALUES LESS THAN (202412),
    PARTITION p202412 VALUES LESS THAN (202501),
    PARTITION p202501 VALUES LESS THAN (202502)
    -- 자동 파티션 추가 스크립트 필요
);
```

**설계 포인트:**
- **월별 파티셔닝**: 3개월 데이터 조회 시 전체 스캔 방지
- **복합 인덱스**: `(user_id, used_at DESC)` - 최근 N개 조회 최적화
- **시간 인덱스**: 3개월 범위 검색 최적화
- **CASCADE 삭제**: 사용자 삭제 시 이력도 자동 삭제

**파티션 관리 배치:**
```sql
-- 매월 1일 새 파티션 추가
DELIMITER $$
CREATE EVENT add_monthly_partition
ON SCHEDULE EVERY 1 MONTH
STARTS '2025-01-01 00:00:00'
DO BEGIN
    SET @next_month = DATE_FORMAT(DATE_ADD(NOW(), INTERVAL 1 MONTH), '%Y%m');
    SET @partition_name = CONCAT('p', @next_month);
    SET @sql = CONCAT('ALTER TABLE password_history ADD PARTITION (PARTITION ', 
                      @partition_name, ' VALUES LESS THAN (', 
                      @next_month + 1, '))');
    PREPARE stmt FROM @sql;
    EXECUTE stmt;
    DEALLOCATE PREPARE stmt;
END$$
DELIMITER ;
```

### 3. password_usage_count 테이블
각 비밀번호 해시가 몇 명에게 사용되고 있는지 집계합니다.

```sql
CREATE TABLE password_usage_count (
    password_hash VARCHAR(255) PRIMARY KEY,
    usage_count INT NOT NULL DEFAULT 1,
    last_updated TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    INDEX idx_usage_count (usage_count),
    INDEX idx_last_updated (last_updated)
);
```

**설계 포인트:**
- **PRIMARY KEY**: password_hash로 중복 방지 및 빠른 조회
- **usage_count 인덱스**: 100개 이상 비밀번호 검색 시 사용
- **동시성 제어**: 낙관적 락 또는 Redis 분산 락 필요

**사용 카운트 업데이트 쿼리:**
```sql
-- 증가
INSERT INTO password_usage_count (password_hash, usage_count) 
VALUES (?, 1) 
ON DUPLICATE KEY UPDATE usage_count = usage_count + 1;

-- 감소
UPDATE password_usage_count 
SET usage_count = GREATEST(usage_count - 1, 0) 
WHERE password_hash = ?;
```

## 데이터 볼륨 예측

### 1억 계정 기준

| 테이블                  | 레코드 수          | 예상 크기  |
| -------------------- | -------------- | ------ |
| users                | 1억             | ~20GB  |
| password_history     | 5억 (평균 5개/사용자) | ~100GB |
| password_usage_count | 1천만 (중복 제거)    | ~2GB   |

**스토리지 전략:**
- `password_history`: 아카이빙 정책 필요 (1년 이상 → Cold Storage)
- SSD 사용 권장 (빈번한 조회)

## 인덱스 전략

### 최근 5개 비밀번호 조회
```sql
SELECT password_hash 
FROM password_history 
WHERE user_id = ? 
ORDER BY used_at DESC 
LIMIT 5;
```
- 사용 인덱스: `idx_user_recent (user_id, used_at DESC)`
- Covering Index로 최적화 가능

### 3개월 이내 비밀번호 조회
```sql
SELECT password_hash 
FROM password_history 
WHERE user_id = ? 
  AND used_at >= DATE_SUB(NOW(), INTERVAL 3 MONTH);
```
- 파티션 프루닝: 최근 3개월 파티션만 스캔
- 인덱스: `idx_user_recent` 활용

### 100개 이상 사용 비밀번호 조회
```sql
SELECT password_hash 
FROM password_usage_count 
WHERE usage_count >= 100;
```
- 사용 인덱스: `idx_usage_count`
- Bloom Filter 재구축 시 활용

## 마이그레이션 전략

### 초기 구축
```java
// 기존 users 테이블에서 password_usage_count 구축
@Service
public class InitialDataMigration {
    
    @Transactional(readOnly = true)
    public void buildPasswordUsageCount() {
        int batchSize = 10000;
        long totalUsers = userRepository.count();
        
        for (long offset = 0; offset < totalUsers; offset += batchSize) {
            Map<String, Integer> countMap = new HashMap<>();
            
            List<User> users = userRepository.findAllWithOffset(offset, batchSize);
            for (User user : users) {
                countMap.merge(user.getCurrentPassword(), 1, Integer::sum);
            }
            
            // Bulk insert
            usageCountRepository.batchInsert(countMap);
            
            log.info("Processed {}/{} users", offset + batchSize, totalUsers);
        }
    }
}
```

### 정합성 검증
```sql
-- 실제 사용자 수와 usage_count 비교
SELECT 
    u.password_hash,
    COUNT(*) as actual_count,
    puc.usage_count as recorded_count
FROM users u
LEFT JOIN password_usage_count puc ON u.current_password = puc.password_hash
GROUP BY u.password_hash
HAVING actual_count != COALESCE(recorded_count, 0);
```
