# 캐싱 전략

## 1만 명 단계
### Redis 캐시 도입
**최우선 과제** - 비용 대비 효과가 가장 큼

### 캐싱 대상
1. **인기 토론방** (TTL: 5분)
2. **활성화된 토론방 리스트** (TTL: 3분)
3. **토론방 상세 정보** (TTL: 10분)

### Spring 구현
```java
@Cacheable(value = "popularRooms", key = "#limit")
public List<Room> getPopularRooms(int limit) {
    return roomRepository.findPopularRooms(limit);
}

@CacheEvict(value = "room", key = "#roomId")
public Room updateRoom(Long roomId, RoomDto dto) {
    // 업데이트 로직
}
```

### 효과
- DB 읽기 쿼리 70-80% 감소 예상
- 응답 속도 10배 개선

## 10만 명 단계
### 캐시 전략 고도화

#### 1. Multi-level Cache
```java
// L1: Local Cache (Caffeine)
@Cacheable(value = "roomLocal", key = "#id")
public Room getRoomFromLocal(Long id) { ... }

// L2: Redis (분산 캐시)
@Cacheable(value = "roomRedis", key = "#id")
public Room getRoomFromRedis(Long id) { ... }
```

#### 2. Cache Stampede 방지
```java
RLock lock = redisson.getLock("room:" + roomId);
try {
    if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
        // DB 조회 후 캐싱
        Room room = roomRepository.findById(roomId);
        redisTemplate.opsForValue().set(key, room, 10, TimeUnit.MINUTES);
    }
} finally {
    lock.unlock();
}
```

#### 3. Cache Warming
- 인기 토론방 주기적 사전 로딩
- Scheduler로 새벽 시간 갱신

## 1000만 명 단계
### Redis Cluster 구축
```
Master 1 (Slot 0-5460)    → Slave 1
Master 2 (Slot 5461-10922) → Slave 2
Master 3 (Slot 10923-16383) → Slave 3
```

### 캐시 패턴 다양화
- **Cache Aside**: 일반 조회
- **Write Through**: 실시간 데이터
- **Write Behind**: 통계 데이터 (비동기)

### 캐시 키 설계 전략
```
room:{roomId}                    # 토론방 상세
room:{roomId}:comments           # 댓글 리스트
room:popular:{category}:{limit}  # 카테고리별 인기 토론방
user:{userId}:rooms              # 사용자별 참여 토론방
```

### Eviction Policy
- LRU (Least Recently Used)
- maxmemory-policy: allkeys-lru
- maxmemory: 4GB per instance
