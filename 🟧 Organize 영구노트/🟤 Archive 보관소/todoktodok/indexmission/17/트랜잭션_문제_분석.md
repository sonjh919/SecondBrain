# 쿠폰 발급 트랜잭션 미동작 문제 분석

## 1. 문제 상황

회원에게 쿠폰을 발급하는 과정에서 **트랜잭션이 제대로 동작하지 않는 문제**가 발생했습니다.

### 문제가 발생한 코드 위치
- `MemberCouponController.java:37-38`
- `CouponIssuer.java:31-44`

---

## 2. 코드 흐름 분석

### 2.1 호출 흐름도

```
MemberCouponController.issueCoupon()
  ↓
CouponIssuer.issueCoupon() [@Transactional]
  ↓
  ├─ MemberService.getMember(memberId) [@Transactional(readOnly = true)]
  ├─ CouponService.getCouponWithLock(couponId) [@Transactional]
  ├─ CouponIssuer.issueMemberCoupon() [@Transactional]
  │    ├─ MemberCoupon.issue() [도메인 로직]
  │    └─ MemberCouponRepository.save()
  └─ observers.forEach(observer -> observer.onIssue())
```

### 2.2 현재 코드 구조

#### MemberCouponController.java
```java:src/main/java/coupon/coupon/ui/MemberCouponController.java
@PostMapping
public Long issueCoupon(@RequestBody IssueCouponRequest request) {
    return couponIssuer.issueCoupon(request.couponId(), request.memberId()).getId();
}
```
**문제점**: Controller에 `@Transactional`이 없음

#### CouponIssuer.java
```java:src/main/java/coupon/coupon/service/CouponIssuer.java
@Transactional
public MemberCoupon issueCoupon(Long couponId, Long memberId) {
    Member member = memberService.getMember(memberId);  // readOnly 트랜잭션
    Coupon coupon = couponService.getCouponWithLock(couponId);
    MemberCoupon memberCoupon = issueMemberCoupon(member, coupon);  // 같은 클래스 내 호출
    observers.forEach(observer -> observer.onIssue(memberCoupon.getId()));
    return memberCoupon;
}

@Transactional
public MemberCoupon issueMemberCoupon(Member member, Coupon coupon) {
    MemberCoupon memberCoupon = MemberCoupon.issue(member.getId(), coupon);
    memberCouponRepository.save(memberCoupon);
    return memberCoupon;
}
```

## 3. 트랜잭션이 동작하지 않는 원인

### 🔴 원인 1: 같은 클래스 내부 메서드 호출 시 AOP 프록시 미작동

```java
// CouponIssuer.java:37
MemberCoupon memberCoupon = issueMemberCoupon(member, coupon);
```

**문제점:**
- `issueCoupon()` 메서드 내에서 `issueMemberCoupon()`을 **this**로 호출
- Spring AOP는 **프록시 객체를 통한 외부 호출**에만 적용됨
- 같은 클래스 내부에서 호출 시 `@Transactional`이 **무시됨**

**동작 원리:**
```
외부 → Proxy(CouponIssuer) → 실제 CouponIssuer.issueCoupon()
                                    ↓
                              this.issueMemberCoupon() ← 프록시 미적용!
```

### 🔴 원인 2: readOnly 트랜잭션과 Write 트랜잭션의 중첩

```java
// CouponIssuer.issueCoupon() - @Transactional (write)
Member member = memberService.getMember(memberId);  // @Transactional(readOnly = true)
```

**문제점:**
- `MemberService.getMember()`는 `readOnly = true`로 선언
- 외부 트랜잭션(`issueCoupon`)은 write 트랜잭션
- 트랜잭션 전파 설정에 따라 **readOnly 속성이 충돌**할 수 있음

**발생 가능한 시나리오:**
```
1. issueCoupon() 시작 → Write 트랜잭션 시작
2. getMember() 호출 → readOnly 트랜잭션 참여 시도
3. 트랜잭션 매니저가 readOnly 속성 충돌 감지
4. 예외 발생 또는 트랜잭션 롤백 실패
```

### 🔴 원인 3: Coupon.issue()의 상태 변경이 영속성 컨텍스트에 반영되지 않을 수 있음

```java:src/main/java/coupon/coupon/domain/Coupon.java
public void issue() {
    // ...
    this.issueCount++;  // dirty checking 필요
}
```

**문제점:**
- `Coupon` 엔티티의 `issueCount` 증가
- `CouponService.getCouponWithLock()`이 `@Transactional` 없이 실행되면 영속성 컨텍스트가 별도로 관리될 수 있음
- Dirty Checking이 제대로 동작하지 않을 수 있음

### 🔴 원인 4: CouponService.getCouponWithLock() 트랜잭션 전파 문제

```java:src/main/java/coupon/coupon/service/CouponService.java
@Service
@Transactional  // 클래스 레벨
public class CouponService {

    public Coupon getCouponWithLock(Long couponId) {  // 메서드에 명시 없음
        return couponRepository.findByIdWithPessimisticLock(couponId)
            .orElseThrow(...);
    }
}
```

**문제점:**
- 클래스 레벨에 `@Transactional`이 있지만 메서드마다 별도 설정 없음
- `readOnly` 설정이 없어서 기본값(false) 사용
- 외부 트랜잭션과 전파 방식에 따라 **락 획득 시점과 해제 시점이 불명확**

## 4. 해결 방안

### ✅ 해결 방안 1: 내부 메서드 호출 제거 (권장)

**Before:**
```java
@Transactional
public MemberCoupon issueCoupon(Long couponId, Long memberId) {
    Member member = memberService.getMember(memberId);
    Coupon coupon = couponService.getCouponWithLock(couponId);
    MemberCoupon memberCoupon = issueMemberCoupon(member, coupon);  // ❌
    observers.forEach(observer -> observer.onIssue(memberCoupon.getId()));
    return memberCoupon;
}
```

**After:**
```java
@Transactional
public MemberCoupon issueCoupon(Long couponId, Long memberId) {
    Member member = memberService.getMember(memberId);
    Coupon coupon = couponService.getCouponWithLock(couponId);

    // 내부 메서드 호출 제거, 로직 직접 구현
    MemberCoupon memberCoupon = MemberCoupon.issue(member.getId(), coupon);
    memberCouponRepository.save(memberCoupon);

    observers.forEach(observer -> observer.onIssue(memberCoupon.getId()));
    return memberCoupon;
}
```

**장점:**
- 트랜잭션 경계가 명확해짐
- AOP 프록시 우회 문제 해결
- 불필요한 메서드 제거로 코드 단순화

### ✅ 해결 방안 2: Self Injection 사용

```java
@Component
public class CouponIssuer {

    @Autowired
    @Lazy
    private CouponIssuer self;  // 자기 자신을 주입

    @Transactional
    public MemberCoupon issueCoupon(Long couponId, Long memberId) {
        Member member = memberService.getMember(memberId);
        Coupon coupon = couponService.getCouponWithLock(couponId);
        MemberCoupon memberCoupon = self.issueMemberCoupon(member, coupon);  // ✅ 프록시 통해 호출
        observers.forEach(observer -> observer.onIssue(memberCoupon.getId()));
        return memberCoupon;
    }
}
```

**장점:**
- `@Transactional`이 정상 작동
- 메서드 분리 유지 가능

**단점:**
- 순환 참조 가능성 (`@Lazy` 필수)
- 코드 복잡도 증가

### ✅ 해결 방안 3: MemberService.getMember()의 트랜잭션 설정 변경

**Before:**
```java
@Transactional(readOnly = true)
public Member getMember(Long memberId) {
    return memberRepository.findById(memberId)
        .orElseThrow(...);
}
```

**After (Option 1):**
```java
@Transactional(readOnly = true, propagation = Propagation.SUPPORTS)
public Member getMember(Long memberId) {
    return memberRepository.findById(memberId)
        .orElseThrow(...);
}
```

**After (Option 2):**
```java
// readOnly 제거
@Transactional
public Member getMember(Long memberId) {
    return memberRepository.findById(memberId)
        .orElseThrow(...);
}
```

**설명:**
- `SUPPORTS`: 외부 트랜잭션이 있으면 참여, 없으면 트랜잭션 없이 실행
- Write 트랜잭션과의 충돌 방지

### ✅ 해결 방안 4: 트랜잭션 전파 레벨 명시

```java
@Service
@Transactional
public class CouponService {

    @Transactional(propagation = Propagation.MANDATORY)  // 외부 트랜잭션 필수
    public Coupon getCouponWithLock(Long couponId) {
        return couponRepository.findByIdWithPessimisticLock(couponId)
            .orElseThrow(...);
    }
}
```

**장점:**
- 락 획득이 반드시 외부 트랜잭션 내에서 실행됨을 보장
- 트랜잭션 경계 명확화

## 5. 권장 해결 방법 조합

```java
// CouponIssuer.java
@Component
public class CouponIssuer {

    private final CouponService couponService;
    private final MemberService memberService;
    private final MemberCouponRepository memberCouponRepository;
    private final List<MemberCouponObserver> observers;

    @Transactional
    public MemberCoupon issueCoupon(Long couponId, Long memberId) {
        log.info("쿠폰 발급 요청: couponId={}, memberId={}", couponId, memberId);

        // 1. 회원 조회 (readOnly 트랜잭션이지만 SUPPORTS로 외부 트랜잭션 참여)
        Member member = memberService.getMember(memberId);

        // 2. 쿠폰 조회 및 락 획득 (MANDATORY로 외부 트랜잭션 필수)
        Coupon coupon = couponService.getCouponWithLock(couponId);

        // 3. 회원 쿠폰 발급 (내부 메서드 호출 제거, 로직 직접 구현)
        MemberCoupon memberCoupon = MemberCoupon.issue(member.getId(), coupon);
        memberCouponRepository.save(memberCoupon);

        // 4. 옵저버 패턴으로 부가 작업 수행
        observers.forEach(observer -> observer.onIssue(memberCoupon.getId()));

        log.info("쿠폰 발급 완료: couponId={}, memberId={}, memberCouponId={}",
                 couponId, memberId, memberCoupon.getId());

        return memberCoupon;
    }

    // issueMemberCoupon() 메서드 제거 (불필요)
}
```

```java
// MemberService.java
@Service
@Transactional
public class MemberService {

    @Transactional(readOnly = true, propagation = Propagation.SUPPORTS)
    public Member getMember(Long memberId) {
        return memberRepository.findById(memberId)
            .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 회원입니다."));
    }
}
```

```java
// CouponService.java
@Service
@Transactional
public class CouponService {

    @Transactional(propagation = Propagation.MANDATORY)
    public Coupon getCouponWithLock(Long couponId) {
        return couponRepository.findByIdWithPessimisticLock(couponId)
            .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 쿠폰입니다."));
    }
}
```

## 6. 트랜잭션 전파 레벨 정리

| 전파 레벨 | 설명 | 사용 시나리오 |
|---------|------|-------------|
| **REQUIRED** (기본값) | 외부 트랜잭션이 있으면 참여, 없으면 새로 생성 | 대부분의 일반적인 경우 |
| **SUPPORTS** | 외부 트랜잭션이 있으면 참여, 없으면 트랜잭션 없이 실행 | 조회 메서드가 write 트랜잭션 내에서도 호출될 때 |
| **MANDATORY** | 외부 트랜잭션이 **반드시** 있어야 함, 없으면 예외 | 락 획득이 필수인 경우 |
| **REQUIRES_NEW** | 항상 새로운 트랜잭션 생성, 외부 트랜잭션은 일시 중단 | 독립적인 로깅, 감사 작업 |
| **NOT_SUPPORTED** | 트랜잭션 없이 실행, 외부 트랜잭션이 있으면 일시 중단 | 외부 API 호출 등 |
| **NEVER** | 트랜잭션 없이 실행, 외부 트랜잭션이 있으면 예외 | 트랜잭션이 절대 있으면 안 되는 경우 |

## 8. 결론

### 핵심 문제
1. **같은 클래스 내부 메서드 호출**로 인한 AOP 프록시 미작동
2. **readOnly 트랜잭션과 write 트랜잭션 중첩** 시 충돌
3. **트랜잭션 전파 레벨 미명시**로 인한 불명확한 동작

### 최종 해결책
1. `issueMemberCoupon()` 메서드 호출 제거, 로직을 `issueCoupon()`에 직접 구현
2. `MemberService.getMember()`에 `propagation = SUPPORTS` 추가
3. `CouponService.getCouponWithLock()`에 `propagation = MANDATORY` 추가
4. 로그 설정으로 트랜잭션 동작 검증
5. 테스트 코드로 롤백 동작 확인

### 학습 포인트
- Spring AOP는 **프록시 기반**이므로 내부 메서드 호출 시 적용되지 않음
- 트랜잭션 전파 레벨을 명시하여 **트랜잭션 경계를 명확히** 할 것
- **readOnly 트랜잭션**은 최적화를 위한 것이므로 write 작업과 혼용 주의
- 비관적 락(Pessimistic Lock)은 **반드시 트랜잭션 내**에서 획득해야 함
