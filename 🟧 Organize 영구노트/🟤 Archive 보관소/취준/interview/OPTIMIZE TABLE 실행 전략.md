[](재구성%20성능%20측정%20메트릭.md)### Question
Slack 알림이 발생한 후, 실제로 OPTIMIZE TABLE을 실행할 때 서비스 중단을 방지하기 위해 어떤 전략을 사용하셨나요?

### Answer
단편화 비율 확인 작업은 운영 서버 기준으로 2주에 한번, 새벽 2시에 실시하고 있습니다. 현재는 알림만 받고 실제로 optimize table을 수행하는 단계까지 자동화되어있지는 않은 상태입니다.

### Feedback
**예상 베스트 답변**: 현재는 2주 주기로 새벽 2시에 Fragmentation을 체크하여 Slack 알림을 받고, 실제 OPTIMIZE TABLE은 수동으로 수행하는 반자동화 단계입니다. 수동 수행 시에는 트래픽이 가장 적은 새벽 시간대를 선택하고, 테이블 크기와 예상 소요 시간을 사전에 파악하여 계획적으로 진행합니다. 향후 개선 방향으로는 (1) OPTIMIZE TABLE의 완전 자동화, (2) MySQL 5.6 이상에서 지원하는 Online DDL(ALGORITHM=INPLACE) 활용, (3) 대용량 테이블의 경우 pt-online-schema-change 도구를 활용한 무중단 재구성을 고려하고 있습니다.

**현재 답변 평가**: 모니터링 주기(2주)와 실행 시간(새벽 2시)을 명확히 밝히셨고, 현재 상태를 솔직하게 설명하신 점이 좋습니다. 실무에서는 완전 자동화보다 단계적 접근이 안전할 수 있으며, 알림 기반 수동 처리도 합리적인 선택입니다. 다만 수동으로 OPTIMIZE TABLE을 수행할 때 고려하는 사항(테이블 크기, 예상 소요 시간, 모니터링 방법)이나, 향후 완전 자동화 또는 무중단 처리 방안에 대한 계획을 추가로 언급하면 더욱 완성도 높은 답변이 됩니다.

**추천사항**: "아직 자동화되지 않았다"는 점을 단점이 아닌 "단계적 개선 과정"으로 프레이밍하시고, 실제 수동 수행 시 경험(특정 테이블 재구성 소요 시간, 서비스 영향 여부 등)이나 향후 개선 로드맵을 언급하시면 문제 인식과 개선 의지를 보여줄 수 있습니다.

### Recommendation
현재 TodokTodok 프로젝트의 Index Fragmentation 관리는 **단계적 자동화 접근법**을 취하고 있습니다.

**모니터링 스케줄**은 운영 서버 기준으로 2주에 한 번, 매주 격주 새벽 2시에 Spring Scheduler를 통해 자동으로 실행됩니다. 이 시간대는 서비스 트래픽 분석 결과 DAU(일일 활성 사용자)가 가장 낮은 시간대로, 만약의 성능 저하가 발생해도 사용자 영향을 최소화할 수 있습니다.

**현재 운영 방식**은 반자동화 단계입니다. Scheduler가 `INFORMATION_SCHEMA.TABLES`를 조회하여 Fragmentation Rate가 10% 이상인 테이블을 감지하면 Slack으로 알림을 발송하고, 이후 DBA 또는 백엔드 개발자가 직접 OPTIMIZE TABLE을 수행합니다. 완전 자동화하지 않은 이유는 다음과 같습니다:

1. **안전성 우선**: OPTIMIZE TABLE은 테이블 잠금(Table Lock)이 발생할 수 있어, 예상치 못한 트래픽 증가나 긴급 상황에서 수동 제어가 필요
2. **테이블별 특성 고려**: 대용량 테이블(수백만 건 이상)과 소규모 테이블의 재구성 전략이 다르므로, 상황에 맞는 판단 필요
3. **점진적 개선**: 초기에는 모니터링과 알림 시스템 구축에 집중하고, 충분한 운영 데이터를 확보한 후 자동화 단계로 발전

**수동 수행 시 고려사항**:
- 테이블 크기와 예상 소요 시간을 사전 파악 (`SHOW TABLE STATUS` 활용)
- Grafana로 실시간 데이터베이스 커넥션 수, 쿼리 응답 시간 모니터링
- 중요 비즈니스 로직(결제, 인증 등)과 관련 없는 테이블부터 우선 처리

**향후 개선 계획**:
1. **Online DDL 적용**: MySQL 5.6+ 환경에서 `ALTER TABLE ... ALGORITHM=INPLACE` 또는 InnoDB의 Online DDL 기능 활용하여 테이블 잠금 없이 재구성
2. **pt-online-schema-change 도구 검토**: 대용량 테이블의 경우 Percona Toolkit의 pt-osc를 활용하여 청크 단위 무중단 재구성
3. **완전 자동화**: 충분한 운영 경험이 쌓이면 소규모 테이블부터 자동 OPTIMIZE TABLE 수행
4. **재구성 효과 측정**: 재구성 전후 쿼리 성능, 디스크 I/O, 응답 시간을 자동으로 수집하여 개선 효과를 정량화

이러한 단계적 접근을 통해 서비스 안정성을 유지하면서도 지속적으로 데이터베이스 성능을 최적화하고 있습니다.
