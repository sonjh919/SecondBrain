### Question
재구성 전후 성능을 측정하기 위해 어떤 메트릭을 추적하는지 말씀해주세요.

### Answer
파편화 비율이 10%가 넘게 된다면 WARN 레벨로 로그가 찍히는데, 이 로그를 탐지해서 slack 알람을 보냅니다.

### Feedback
**예상 베스트 답변**: 재구성 전후 성능 측정을 위해 다음과 같은 메트릭을 추적합니다. (1) **Fragmentation Rate**: `data_free / (data_length + index_length)` 비율이 재구성 후 실제로 감소했는지 확인, (2) **쿼리 응답 시간**: Slow Query Log나 APM 도구를 통해 해당 테이블을 사용하는 쿼리의 평균/최대 응답 시간 개선 여부 측정, (3) **디스크 I/O**: Grafana + Prometheus를 통해 테이블 스캔 시 발생하는 물리적 읽기(Physical Reads) 감소 확인, (4) **테이블 크기**: `data_length + index_length` 값이 재구성 후 줄어들었는지 확인하여 공간 효율성 개선 검증. 이러한 지표들을 재구성 전 1주일, 재구성 후 1주일 동안 수집하여 비교 분석합니다.

**현재 답변 평가**: Slack 알림 시스템 구축에 대해 명확히 설명하셨지만, 질문은 "재구성 전후 성능 측정"에 관한 것이었습니다. 현재 답변은 Fragmentation 감지 시스템에 초점이 맞춰져 있어, 실제로 OPTIMIZE TABLE을 수행한 후 성능이 개선되었는지 확인하는 방법에 대한 내용이 빠져있습니다. 재구성의 효과를 정량적으로 측정하는 메트릭(쿼리 응답 시간, 디스크 I/O, 테이블 크기 등)을 추가로 언급하시면 좋습니다.

**추천사항**: Loki, Prometheus, Grafana 같은 모니터링 도구를 이력서에 언급하셨으니, 이들을 활용하여 재구성 전후의 성능 변화를 어떻게 추적하고 시각화하는지 구체적으로 설명하시면 더욱 설득력 있는 답변이 됩니다. 예를 들어 "Grafana 대시보드에서 특정 테이블의 평균 쿼리 시간이 재구성 후 X초에서 Y초로 개선되었다"는 식의 정량적 데이터를 제시하면 좋습니다.

### Recommendation
재구성 전후 성능 측정은 OPTIMIZE TABLE의 실제 효과를 검증하고 향후 개선 방향을 결정하는 중요한 과정입니다. TodokTodok 프로젝트에서는 다음과 같은 다층적 메트릭 추적 체계를 구축했습니다.

**1. Fragmentation Rate 변화 추적**

재구성의 가장 직접적인 효과는 Fragmentation Rate 감소입니다. `INFORMATION_SCHEMA.TABLES`를 통해 다음 지표를 재구성 전후로 비교합니다:

```sql
SELECT 
    table_name,
    ROUND(data_free / 1024 / 1024, 2) as data_free_mb,
    ROUND((data_length + index_length) / 1024 / 1024, 2) as total_size_mb,
    ROUND((data_free / (data_length + index_length)) * 100, 2) as fragmentation_rate
FROM information_schema.TABLES
WHERE table_schema = 'todoktodok' 
  AND table_name = 'target_table';
```

예상 결과: 재구성 전 15% → 재구성 후 1% 미만으로 감소

**2. 쿼리 응답 시간 측정**

실제 비즈니스 로직에서 체감할 수 있는 성능 개선을 측정하기 위해 다음 방법을 사용합니다:

- **Slow Query Log 분석**: MySQL의 `slow_query_log`를 활성화하여 해당 테이블을 사용하는 쿼리들의 실행 시간 변화 추적
- **APM 도구 활용**: 애플리케이션 레벨에서 특정 API의 응답 시간 모니터링
- **비교 기간**: 재구성 전 1주일과 재구성 후 1주일의 평균/P95/P99 응답 시간 비교

예시: Full-Text 검색 쿼리 평균 응답 시간 18.6초 → 10초 (46% 개선)

**3. 디스크 I/O 메트릭**

Prometheus + Grafana를 활용한 시스템 레벨 모니터링:

- **Physical Reads**: 디스크에서 실제로 읽어들이는 데이터 양
- **Buffer Pool Hit Rate**: InnoDB 버퍼 풀의 캐시 히트율
- **IOPS (I/O Operations Per Second)**: 초당 I/O 작업 수

Fragmentation이 해소되면 연속된 데이터 블록 읽기가 가능해져 Physical Reads가 감소하고 Buffer Pool 효율이 증가합니다.

**4. 테이블 물리적 크기 변화**

```sql
SELECT 
    table_name,
    ROUND((data_length + index_length) / 1024 / 1024, 2) as size_mb_before
FROM information_schema.TABLES
WHERE table_name = 'target_table';

-- OPTIMIZE TABLE 수행 후

SELECT 
    table_name,
    ROUND((data_length + index_length) / 1024 / 1024, 2) as size_mb_after
FROM information_schema.TABLES
WHERE table_name = 'target_table';
```

Fragmentation 제거로 불필요한 공간이 회수되어 테이블 크기가 10-20% 감소하는 효과를 기대할 수 있습니다.

**5. 통합 모니터링 대시보드**

Grafana 대시보드에서 다음 패널들을 구성하여 한눈에 비교:
- Fragmentation Rate 추이 그래프
- 주요 쿼리 응답 시간 추이 (재구성 시점에 수직선 표시)
- 디스크 I/O 메트릭 추이
- 테이블 크기 변화

**6. 알림 시스템 연계**

현재 구축된 Slack 알림 시스템은 Fragmentation이 10% 이상일 때 WARN 레벨 로그를 탐지하여 알림을 발송합니다. 이를 확장하여:
- 재구성 작업 시작/완료 알림
- 재구성 전후 주요 메트릭 비교 리포트 자동 발송
- 예상보다 성능 개선이 미미한 경우 추가 분석 필요 알림

이러한 다각도의 메트릭 추적을 통해 재구성 작업의 실제 효과를 정량적으로 검증하고, 지속적인 데이터베이스 최적화 전략을 수립할 수 있습니다.
