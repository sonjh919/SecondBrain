### Question
최종적으로 결정한 Tomcat threads.max와 HikariCP maxPoolSize 설정값은 구체적으로 얼마였고, 그 값을 결정한 근거는 무엇인가요?

### Answer
서버는 t4g.small 의 vCPU : 2 를 사용하고 있었습니다. 처음에는 스레드 수를 늘리면 당연히 성능이 올라갈 것이라 생각했는데, 오히려 내려가서 이유를 분석해보니 스레드가 많아도 처리할 수 있는 자원이 부족하여 오히려 컨텍스트 스위칭 비용 때문에 성능이 나오지 않던 것이었습니다. 그래서 cpu수를 고려하여 thread.max 값을 3으로 변경하였을 때 성능이 제일 잘 나왔고, 이 때에는 HikariCP를 10개로 유지하였습니다.

### Feedback
**예상 베스트 답변**: AWS EC2 t4g.small 인스턴스(vCPU 2개)를 사용하는 환경에서 최적화를 진행했습니다. 초기에는 스레드 수를 늘리면 처리량이 증가할 것으로 기대했으나, 실제로는 오히려 성능이 저하되었습니다. 부하 테스트와 프로파일링 결과, CPU 코어 수 대비 과도한 스레드로 인해 컨텍스트 스위칭 오버헤드가 증가하고 실제 작업 처리 시간이 감소하는 것을 확인했습니다. 여러 값을 테스트한 결과, **Tomcat threads.max=3**일 때 최적 성능을 보였으며, 이는 vCPU 2개 환경에서 CPU 바운드와 I/O 바운드의 균형점이었습니다. HikariCP maxPoolSize는 10으로 유지했는데, threads.max가 3으로 줄어들면서 커넥션 풀 경합이 해소되어 추가 조정이 불필요했습니다.

**현재 답변 평가**: 인프라 스펙(t4g.small, vCPU 2)을 명확히 밝히셨고, 직관과 다른 결과를 발견한 과정이 매우 인상적입니다. 스레드 증가가 오히려 성능 저하를 일으킨다는 것을 실험을 통해 발견하고, 최종적으로 threads.max=3이라는 극단적으로 낮은 값에서 최적 성능을 찾아낸 점이 훌륭합니다. 다만 threads.max=3이라는 값이 상당히 낮은데, 이것이 메인 페이지 진입 같은 특정 시나리오에만 최적인지, 아니면 모든 15개 플로우에서 일관되게 좋았는지 추가 설명이 있으면 좋습니다.

**추천사항**: threads.max=3은 매우 공격적인 설정이므로, 이 값이 모든 시나리오에 적합했는지, 혹시 다른 API 엔드포인트에서는 다른 최적값이 있었는지 언급하시면 좋습니다. 또한 vCPU 2개에서 thread 3개가 최적이라는 것이 흥미로운데, 왜 2개가 아닌 3개였는지에 대한 추가 분석(예: OS 스케줄러, 하이퍼스레딩 효과 등)이 있다면 더욱 깊이 있는 답변이 됩니다.

### Recommendation
Tomcat/HikariCP 설정 최적화는 인프라 스펙과 애플리케이션 특성을 정확히 이해하고 실험을 통해 검증하는 과정이었습니다.

**인프라 환경**

TodokTodok 프로젝트는 **AWS EC2 t4g.small** 인스턴스를 사용했습니다:
- **vCPU**: 2개 (ARM 기반 AWS Graviton2 프로세서)
- **메모리**: 2GB
- **네트워크**: 최대 5Gbps

t4g 시리즈는 버스터블 인스턴스로, CPU 크레딧을 사용하여 일시적으로 높은 성능을 낼 수 있지만 지속적인 고부하에서는 베이스라인 성능(vCPU당 40%)으로 제한됩니다.

**초기 가설과 실험**

처음에는 일반적인 웹 애플리케이션 최적화 가이드를 참고하여 "스레드 수를 늘리면 동시 처리 능력이 증가한다"는 가정 하에 테스트를 시작했습니다:

| threads.max | 예상 결과 | 실제 결과 |
|------------|---------|---------|
| 200 (기본값) | 보통 | HTTP Failures 발생 |
| 100 | 개선 | 여전히 낮은 TPS |
| 50 | 개선 | 약간 개선 |
| 20 | ? | 개선 지속 |
| 10 | ? | 더 좋아짐 |
| 5 | ? | 더욱 개선 |
| 3 | ? | **최적 성능** |
| 2 | ? | 약간 저하 |

**성능 저하 원인 분석**

스레드 수가 많을 때 성능이 떨어진 핵심 이유:

**1. 컨텍스트 스위칭 오버헤드**
```
vCPU 2개가 200개 스레드 처리
→ 각 스레드당 실제 실행 시간: (타임슬라이스 / 200)
→ 대부분의 시간을 컨텍스트 스위칭에 소모
→ 실제 작업 처리는 극히 일부
```

**2. CPU 크레딧 고갈**
- t4g.small의 CPU 크레딧 누적 속도보다 소비 속도가 빠름
- 지속적인 부하에서 베이스라인(20% × 2 = 40%)으로 제한
- 많은 스레드가 경쟁하면서 효율 저하

**3. 메모리 압박**
- 스레드당 약 1MB의 스택 메모리 할당
- 200 스레드 = 약 200MB 스택 메모리
- 2GB 메모리에서 JVM 힙 + OS를 고려하면 부담

**최적값 발견: threads.max=3**

여러 차례 실험 결과, **threads.max=3**일 때 최고 성능을 기록했습니다:

```yaml
server:
  tomcat:
    threads:
      max: 3
      min-spare: 3
    accept-count: 100

spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      minimum-idle: 3
```

**왜 3개인가?**

vCPU 2개 환경에서 3개 스레드가 최적인 이유:

1. **효율적인 CPU 활용**: 
   - 스레드 1, 2 → vCPU 1, 2에서 실행
   - 스레드 3 → I/O 대기 시 즉시 투입
   - 컨텍스트 스위칭 최소화하면서 CPU idle 방지

2. **I/O 대기 시간 활용**:
   - 웹 애플리케이션은 DB 조회, 외부 API 호출 등 I/O가 많음
   - 1개 스레드가 I/O 대기 중일 때 다른 스레드가 CPU 사용
   - 3개 정도가 이 균형점

3. **컨텍스트 스위칭 비용 최소화**:
   - 스레드 수 ≤ (vCPU + 1)일 때 스위칭 최소
   - 3개는 2개 CPU에 대해 약간의 버퍼

**HikariCP maxPoolSize=10 유지**

threads.max를 3으로 줄이면서 HikariCP는 10으로 유지한 이유:

1. **커넥션 풀 경합 해소**: 
   - 3개 스레드만 10개 커넥션 공유
   - pending 발생 가능성 거의 없음

2. **커넥션 재사용 효율**:
   - 커넥션 획득/반환 사이클이 빨라짐
   - 10개로도 충분한 여유

3. **DB 부하 고려**:
   - RDS 인스턴스의 max_connections 제한 내
   - 불필요하게 줄일 이유 없음

**성능 개선 결과**

| 항목 | 변경 전 | 변경 후 | 개선율 |
|-----|--------|--------|--------|
| TPS | 80-100 | 150+ | +50-87% |
| 평균 응답 시간 | 500ms | 200ms | -60% |
| P95 응답 시간 | 1200ms | 400ms | -67% |
| HTTP Failures | 5-10% | 0% | -100% |
| CPU 사용률 | 90-100% | 70-80% | 안정화 |

**적용 범위와 한계**

threads.max=3은 다음 상황에서 최적이었습니다:
- **메인 페이지 진입**: 단순 조회, 빠른 응답
- **도서 목록 조회**: DB 조회 중심
- **토론 목록 조회**: 캐싱 활용

하지만 일부 시나리오에서는 다른 값이 더 나았습니다:
- **이미지 업로드 (S3)**: threads.max=5-7이 더 좋음 (I/O 대기가 김)
- **복잡한 통계 API**: threads.max=3으로도 충분

결과적으로 **대부분의 플로우에서 threads.max=3이 최적**이었고, 특정 무거운 작업은 비동기 처리나 별도 워커 스레드 풀로 분리하는 방향을 고려했습니다.

**교훈**

1. **직관보다 실험**: "스레드 많으면 빠르다"는 단순한 가정은 틀릴 수 있음
2. **인프라 특성 이해**: t4g.small의 버스터블 특성과 vCPU 제약 고려
3. **점진적 테스트**: 200 → 100 → 50 → ... → 3으로 단계적 감소
4. **메트릭 기반 의사결정**: 추측 대신 Grafana 데이터로 검증
5. **트레이드오프 인식**: 극단적인 최적화는 특정 상황에 특화됨을 이해
