[](배포%20실패%20감지와%20롤백%20전략.md)### Question
무중단 배포를 구축하게 된 이유는 무엇이고, Rolling Update 방식을 선택한 이유는 무엇인가요?

### Answer
프로젝트 초기에는 단일 EC2 인스턴스로 배포했는데, 새 버전을 배포할 때마다 서비스가 중단되어 사용자들이 접속할 수 없는 문제가 있었습니다.
Rolling Update 방식을 선택한 이유는 ALB 뒤에 여러 EC2 인스턴스를 두고, 인스턴스를 하나씩 순차적으로 업데이트하면서 나머지 인스턴스가 트래픽을 처리하도록 해서 서비스 중단 없이 배포할 수 있기 때문입니다.
Blue-Green 방식도 고려했지만, 학습 프로젝트 특성상 동일한 환경을 두 세트 유지하는 것은 비용 부담이 컸고, Rolling Update는 기존 인스턴스를 재사용하면서도 무중단 배포를 충분히 달성할 수 있어서 더 적합하다고 판단했습니다.

### Feedback
문제 인식, 해결 방법, 그리고 다른 배포 전략(Blue-Green)과의 비교를 통한 의사결정 과정을 매우 명확하게 설명하셨습니다. 다만, 구체적으로 GitHub Actions 워크플로우를 어떻게 구성했는지, ALB의 헬스 체크를 어떻게 활용했는지, 그리고 Rolling Update 과정에서 발생할 수 있는 문제(예: 구버전과 신버전이 동시에 실행되는 상황)를 어떻게 처리했는지 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
**무중단 배포 구축 배경:**

프로젝트 초기에는 단일 EC2 인스턴스에서 애플리케이션을 실행했습니다. 새 버전을 배포할 때마다 다음과 같은 과정이 필요했습니다:

1. 실행 중인 애플리케이션 종료
2. 새 버전 코드 배포
3. 애플리케이션 재시작

이 과정에서 짧게는 30초, 길게는 2-3분의 서비스 중단이 발생했고, 사용자들은 배포 시간대에 서비스에 접속할 수 없었습니다. 특히 테스트와 버그 수정으로 인해 하루에도 여러 번 배포가 발생하는 개발 단계에서는 이러한 서비스 중단이 큰 문제였습니다.

따라서 서비스 가용성을 보장하고 사용자 경험을 개선하기 위해 무중단 배포 환경이 필수적이라고 판단했습니다.

---

**Rolling Update 방식 선택 이유:**

무중단 배포를 위해 여러 배포 전략을 검토했습니다:

**1. Blue-Green 배포:**
- 장점: 완전히 새로운 환경으로 전환되므로 안전하고, 문제 발생 시 즉시 롤백 가능
- 단점: 동일한 환경을 두 세트 유지해야 하므로 **리소스가 2배**로 필요
- 결론: 학습 프로젝트 특성상 EC2 인스턴스, RDS, 기타 리소스를 이중으로 구성하는 것은 비용 부담이 너무 컸습니다

**2. Canary 배포:**
- 장점: 일부 트래픽만 신버전으로 라우팅하여 점진적 검증 가능
- 단점: 복잡한 트래픽 라우팅 설정 필요, 모니터링 및 자동화 구성 복잡
- 결론: 프로젝트 규모와 팀의 인프라 관리 역량 대비 오버엔지니어링이라고 판단

**3. Rolling Update (선택):**
- 장점: 
  - 기존 인스턴스를 재사용하므로 **추가 비용 없음**
  - ALB가 헬스 체크를 통해 자동으로 트래픽 제어
  - 구현이 상대적으로 간단하면서도 무중단 배포 달성
- 단점: 
  - 배포 중 구버전과 신버전이 동시 실행됨
  - Blue-Green 대비 롤백이 느림
- 결론: 프로젝트의 요구사항과 리소스를 고려했을 때 **가장 효율적**

---

**Rolling Update 구현 방식:**

ALB(Application Load Balancer) 뒤에 2대의 EC2 인스턴스를 Target Group으로 구성했습니다. 배포 과정은 다음과 같습니다:

1. **인스턴스 1 배포 시작:**
   - GitHub Actions가 인스턴스 1을 ALB Target Group에서 Drain 상태로 전환
   - 새로운 연결은 인스턴스 2로만 라우팅되고, 기존 연결은 완료될 때까지 대기
   - Drain 완료 후 인스턴스 1에 새 버전 배포 및 재시작
   - ALB 헬스 체크가 통과되면 다시 트래픽 수신 시작

2. **인스턴스 2 배포:**
   - 인스턴스 1이 정상 동작하면, 동일한 방식으로 인스턴스 2 배포
   - 이때는 인스턴스 1이 모든 트래픽 처리

3. **배포 완료:**
   - 모든 인스턴스가 신버전으로 업데이트되고 정상 동작

---

**구버전/신버전 동시 실행 문제 해결:**

Rolling Update의 특성상 배포 중에는 구버전과 신버전이 동시에 실행됩니다. 이로 인해 발생할 수 있는 문제를 다음과 같이 처리했습니다:

1. **API 호환성 유지**: 
   - 버전 간 Breaking Change를 최소화
   - API 변경 시 하위 호환성을 고려한 설계

2. **데이터베이스 마이그레이션 전략**:
   - Flyway를 사용한 DB 스키마 버전 관리
   - 배포 전 먼저 DB 마이그레이션을 실행하여 구버전과 신버전 모두 호환되도록 설계
   - 예: 컬럼 추가 시 NULL 허용하거나 기본값 설정

3. **세션 관리**:
   - Stateless 설계를 기본으로 하여 인스턴스 간 세션 불일치 문제를 원천 차단

이러한 방식으로 비용 효율적이면서도 서비스 중단 없는 배포 환경을 성공적으로 구축할 수 있었습니다.
