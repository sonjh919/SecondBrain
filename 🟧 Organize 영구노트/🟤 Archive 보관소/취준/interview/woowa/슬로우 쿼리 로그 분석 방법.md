### Question
슬로우 쿼리 로그를 어떻게 수집하고 분석하셨나요? 그리고 어떤 기준으로 병목 API를 식별하셨나요?

### Answer
mysql 슬로우 쿼리 로그를 활성화하였고, 3초 이상 소요되는 쿼리를 기록하여 확인하였습니다.

### Feedback
MySQL 슬로우 쿼리 로그 활성화와 3초 기준을 명확히 제시하셨습니다. 다만, 구체적으로 어떤 설정(`long_query_time`, `slow_query_log` 등)을 변경했는지, 로그 파일을 어떻게 분석했는지(수동 확인, 분석 도구 사용 등), 그리고 수집된 슬로우 쿼리 중에서 우선순위를 정하는 기준(빈도, 실행 시간, 비즈니스 영향도 등)을 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
병목 API를 식별하고 성능을 개선하기 위해 MySQL의 슬로우 쿼리 로그 기능을 활용했습니다.

**슬로우 쿼리 로그 설정:**

먼저 MySQL 설정 파일(my.cnf 또는 my.ini)에서 슬로우 쿼리 로그를 활성화했습니다:

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 3;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow-query.log';
```

- `slow_query_log = 'ON'`: 슬로우 쿼리 로그 기능 활성화
- `long_query_time = 3`: 3초 이상 소요되는 쿼리를 슬로우 쿼리로 기록
- `slow_query_log_file`: 로그 파일 저장 경로 지정

3초 기준을 선택한 이유는, 사용자 경험 관점에서 3초 이상의 응답 시간은 명백히 문제가 있다고 판단했기 때문입니다. 또한 초기 분석 단계에서는 명확한 병목 지점을 파악하기 위해 비교적 높은 임계값을 설정했고, 이후 점진적으로 낮춰가며 세밀하게 최적화했습니다.

**로그 분석 방법:**

슬로우 쿼리 로그가 수집되면, 다음과 같은 방법으로 분석했습니다:

1. **로그 파일 직접 확인**: 개발 환경에서는 로그 파일을 직접 열어 쿼리 패턴을 확인했습니다.

2. **mysqldumpslow 도구 활용**: MySQL이 제공하는 `mysqldumpslow` 유틸리티를 사용하여 슬로우 쿼리를 요약하고 정렬했습니다:
   ```bash
   mysqldumpslow -s t -t 10 /var/log/mysql/slow-query.log
   ```
   - `-s t`: 실행 시간 기준 정렬
   - `-t 10`: 상위 10개 쿼리만 표시

3. **Grafana 대시보드**: Prometheus와 MySQL Exporter를 통해 슬로우 쿼리 발생 빈도와 패턴을 시각화하여 모니터링했습니다.

**병목 API 식별 기준:**

수집된 슬로우 쿼리들 중에서 최적화 우선순위를 정하기 위해 다음 기준을 적용했습니다:

1. **실행 시간**: 가장 오래 걸리는 쿼리 (예: 30초)
2. **실행 빈도**: 자주 실행되는 쿼리 (예: 분당 수십 회)
3. **비즈니스 영향도**: 핵심 기능(토론방 검색, 목록 조회 등)과 직결된 쿼리
4. **개선 가능성**: 인덱스 추가나 쿼리 구조 변경으로 개선이 명확한 경우

예를 들어, 토론방 목록 조회 API의 30초 슬로우 쿼리는 실행 시간이 길고, 사용자가 자주 접근하는 핵심 기능이며, 쿼리 구조 개선으로 최적화가 가능하다고 판단하여 최우선 개선 대상으로 선정했습니다. 반면 관리자 통계 페이지의 5초 쿼리는 실행 빈도가 낮아 우선순위를 낮게 설정했습니다.

이러한 체계적인 분석을 통해 실제로 사용자 경험에 영향을 미치는 병목 지점을 정확히 식별하고, 효과적인 최적화 작업을 진행할 수 있었습니다.
