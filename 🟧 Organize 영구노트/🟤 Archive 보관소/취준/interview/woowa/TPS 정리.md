### 시나리오 설정
|**옵션**|**의미**|**단위 / 예시**|
|---|---|---|
|**vus**|동시에 테스트에 참여할 가상 유저 수(Virtual Users)|50명|
|**duration**|테스트 지속 시간|'30s', '5m', '1h'|
|**iterations**|전체 반복 횟수(시간 대신 사용 가능)|100, 500 등|
|**stages**|시간에 따른 단계별 부하(Ramp-up/Ramp-down)|(예: 점진적 증가)|
|**thresholds**|SLA / SLO 기준 설정 (테스트 통과 조건)|95%<500ms 등|
|**rps**|초당 요청 수 제한|rps: 100|
|**gracefulStop**|테스트 종료 시 대기 시간|'10s'|
|**tags / setup / teardown**|시나리오별 메타데이터, 초기화/정리 단계|—|

stages + duration + target : 일정 시간(duration) 동안 target까지 사용자를 점진적으로 늘리며 요청
### 단일 테스트: 메인화면 조회
```javascript
export const options = {
  stages: [
    { duration: '10m', target: 500 }, // 증가
    { duration: '10m', target: 500 } // 유지
  ]
};
```

이 TPS 실험은 시스템의 병목 지점을 찾고 최적화하는 과정이었습니다.

**실험 과정:**  vus 300명부터 시작해 점진적으로 동시 접속자를 늘리며 병목 지점을 파악했습니다. 500명에서 응답시간이 급증했고, 1000명에서는 HikariCP 커넥션풀 포화로 요청 실패가 발생했습니다.

**주요 최적화:** 첫 번째로 Tomcat 스레드를 200에서 30으로, 최종적으로는 3으로 대폭 줄였습니다. 과도한 스레드가 컨텍스트 스위칭으로 CPU를 94%까지 올렸기 때문입니다. 스레드를 줄이자 CPU 사용률이 크게 안정화됐습니다.

두 번째로 Nginx의 keep-alive 커넥션 재사용을 활성화했습니다. Tomcat의 최대 연결 수 제한으로 'Connection reset' 에러가 발생했는데, keep-alive로 커넥션을 재사용하면서 CPU 사용률이 25%까지 떨어지고 안정적인 상태가 됐습니다.

**최종 결과:** 메인화면 단일 조회 테스트를 통해 서버 1대가 안정적으로 처리할 수 있는 최대 동시 접속자는 500명, 목표 TPS는 170~190으로 설정했습니다. Tomcat은 최소 스레드 3개로도 충분했고, DB가 병목이 아니었기 때문에 HikariCP는 기본값 10으로 충분히 여유가 있었습니다. 적절한 스레드풀 크기와 커넥션 재사용이 성능 최적화의 핵심이었습니다.

### 가중치 테스트: 
15개의 핵심 서비스 사용 플로우를 정하고, 각 플로우에 가중치를 다르게 주어 트래픽을 분배해 보냅니다. 이때 접속할 확률이 높고 핵심인 플로우일수록 가중치를 높게 주었습니다.

유저 테스트를 통해 유저가 일반적으로 앱을 키고 사용할 때의 플로우를 뽑아서 진행했습니다. 예를 들면 토론방 검색, 토론방 생성 및 댓글 생성, 좋아요 등이 있습니다.

**실험 과정:** 단일 조회 테스트와 달리, 300명에서도 TPS가 83으로 매우 낮고 3초 이상 지연이 발생했습니다. 이후 적절한 스레드 수와 커넥션풀 크기를 찾기 위해 추가 테스트를 진행했습니다. Tomcat 스레드를 50개로 늘렸을 때 오히려 TPS가 10으로 떨어지고 latency가 5~15초로 악화됐습니다.
테스트를 계속 진행한 결과 서버 1대가 안정적으로 처리할 수 있는 tps는 최대 동시 접속자 300을 기준으로 100이었고, 이때 Tomcat 스레드 10개, HikariCP 5개가 최적값이었습니다.
