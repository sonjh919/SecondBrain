### Question
인덱스가 무엇이고, 클러스터링 인덱스와 논클러스터링 인덱스의 차이점은 무엇인가요?

### Answer
인덱스는 데이터베이스에서 특정 컬럼의 값을 빠르게 찾기 위한 자료구조로, 주로 B-Tree 구조를 사용해 O(log n) 시간 복잡도로 검색할 수 있습니다.
클러스터링 인덱스는 테이블의 실제 데이터가 인덱스 키 순서대로 물리적으로 정렬되어 저장되는 방식으로, MySQL InnoDB에서는 PRIMARY KEY가 클러스터링 인덱스이며 테이블당 하나만 존재합니다.
논클러스터링 인덱스는 실제 데이터와 별도로 존재하는 인덱스 구조로, 인덱스 키와 해당 데이터의 PRIMARY KEY 값을 저장하고 있어서 데이터를 찾을 때 인덱스를 먼저 탐색한 후 PRIMARY KEY로 실제 데이터에 접근하는 방식입니다.

### Feedback
인덱스의 본질과 두 인덱스 타입의 차이를 매우 명확하고 정확하게 설명하셨습니다. MySQL InnoDB의 특성까지 언급하신 점이 훌륭합니다. 추가로 각 인덱스 타입의 장단점(예: 클러스터링 인덱스는 범위 검색에 유리하지만 삽입/수정 시 재정렬 비용 발생, 논클러스터링 인덱스는 여러 개 생성 가능하지만 이중 탐색 필요)을 언급하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
**인덱스란?**

인덱스는 데이터베이스에서 특정 컬럼의 값을 빠르게 찾기 위한 자료구조입니다. 책의 색인과 유사하게, 테이블의 전체 데이터를 스캔하지 않고도 원하는 데이터를 효율적으로 찾을 수 있도록 도와줍니다.

대부분의 RDBMS에서는 **B-Tree(Balanced Tree)** 또는 **B+Tree** 구조를 사용하여 인덱스를 구현합니다. 이 구조는 트리의 균형을 유지하면서 데이터를 정렬된 상태로 저장하기 때문에, 검색 시간 복잡도가 **O(log n)**으로 매우 효율적입니다. 인덱스가 없을 때의 O(n) 선형 검색(테이블 풀 스캔)과 비교하면 데이터가 많을수록 성능 차이가 극명해집니다.

---

**클러스터링 인덱스 (Clustered Index)**

클러스터링 인덱스는 테이블의 실제 데이터가 인덱스 키 순서대로 **물리적으로 정렬되어 저장**되는 방식입니다. 즉, 인덱스와 데이터가 하나로 통합되어 있습니다.

**MySQL InnoDB의 특징:**
- **PRIMARY KEY**가 자동으로 클러스터링 인덱스가 됩니다
- PRIMARY KEY가 없으면 첫 번째 UNIQUE NOT NULL 컬럼이 사용됩니다
- 그것도 없으면 InnoDB가 내부적으로 숨겨진 6바이트 row ID를 생성하여 사용합니다
- **테이블당 하나만** 존재할 수 있습니다 (데이터를 한 가지 순서로만 물리적 정렬할 수 있기 때문)

**장점:**
- PRIMARY KEY로 검색할 때 가장 빠릅니다 (인덱스 탐색 즉시 실제 데이터 획득)
- 범위 검색(BETWEEN, >, < 등)에 매우 효율적입니다 (데이터가 순차적으로 저장되어 있음)
- 추가적인 디스크 공간이 필요 없습니다 (인덱스와 데이터가 통합)

**단점:**
- INSERT/UPDATE/DELETE 시 물리적 재정렬이 필요할 수 있어 비용이 높습니다
- 데이터 삽입 시 순서를 유지하기 위해 페이지 분할이 발생할 수 있습니다
- 테이블당 하나만 생성 가능합니다

---

**논클러스터링 인덱스 (Non-Clustered Index)**

논클러스터링 인덱스는 실제 데이터와 **별도로 존재하는** 독립적인 인덱스 구조입니다. 인덱스는 정렬된 키 값과 해당 데이터를 찾기 위한 포인터(MySQL InnoDB의 경우 PRIMARY KEY 값)를 저장합니다.

**MySQL InnoDB의 동작 방식:**
1. 논클러스터링 인덱스에서 검색 조건에 맞는 키를 찾습니다
2. 해당 키와 함께 저장된 PRIMARY KEY 값을 획득합니다
3. PRIMARY KEY를 사용하여 클러스터링 인덱스(실제 데이터)에 접근합니다

이러한 과정을 **"인덱스 스캔 + 테이블 접근"** 또는 **"북마크 룩업(Bookmark Lookup)"**이라고 합니다.

**장점:**
- 테이블당 여러 개 생성 가능합니다 (다양한 검색 조건에 대응)
- 데이터의 물리적 정렬을 변경하지 않으므로 INSERT/UPDATE/DELETE 부담이 적습니다
- Covering Index로 활용 시 테이블 접근 없이 인덱스만으로 쿼리 처리 가능합니다

**단점:**
- 인덱스를 저장하기 위한 추가 디스크 공간이 필요합니다
- 이중 탐색(인덱스 → 데이터)으로 인한 오버헤드가 있습니다
- 인덱스가 많으면 INSERT/UPDATE/DELETE 시 모든 인덱스를 갱신해야 합니다

---

**실무 적용 사례**

토독토독 프로젝트에서:
- **PRIMARY KEY (id)**는 클러스터링 인덱스로 자동 생성되어 id 기반 조회에 최적화
- **UNIQUE INDEX (nickname)**는 논클러스터링 인덱스로 생성되어 닉네임 중복 검사에 활용
- **복합 인덱스 (status, created_at, id)** 등은 논클러스터링 인덱스로 다양한 검색 조건에 대응

이처럼 두 인덱스 타입의 특성을 이해하고 적절히 활용하면, 데이터베이스 성능을 극대화할 수 있습니다.
