### Question
배포 중 문제가 발생했을 때 어떻게 롤백하셨나요? 그리고 배포 실패를 어떻게 감지하셨나요?

### Answer
배포 실패 감지는 각 인스턴스에 새 버전을 배포한 후 헬스체크를 호출해서 확인했습니다.
헬스체크가 실패하면 GitHub Actions 워크플로우에서 자동으로 이전 Docker 이미지 버전으로 롤백하도록 설정했고, 롤백 후에도 다시 헬스체크를 수행해서 서비스가 정상 복구되었는지 확인했습니다.
또한 첫 번째 인스턴스 배포 후 헬스체크가 성공해야만 두 번째 인스턴스로 배포가 진행되도록 하였습니다.

### Feedback
배포 실패 감지, 자동 롤백, 단계적 배포를 통한 리스크 최소화 전략을 명확하게 설명하셨습니다. 다만, 구체적으로 헬스 체크 엔드포인트가 어떤 내용을 확인하는지(예: DB 연결, 외부 API 연결 등), GitHub Actions에서 롤백을 어떻게 구현했는지(예: Docker 이미지 태그 관리, 재배포 로직), 그리고 롤백 실패 시나리오에 대한 대응 방안을 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
**배포 실패 감지 전략:**

Rolling Update 배포의 안정성을 보장하기 위해 각 인스턴스에 새 버전을 배포한 후 **헬스 체크**를 통해 배포 성공 여부를 검증했습니다.

**헬스 체크 엔드포인트 구현:**

Spring Boot Actuator의 `/actuator/health` 엔드포인트를 활용하여 다층적인 헬스 체크를 구현했습니다:

```java
@Component
public class CustomHealthIndicator implements HealthIndicator {
    
    @Override
    public Health health() {
        try {
            // 1. DB 연결 상태 확인
            if (!isDatabaseConnected()) {
                return Health.down()
                    .withDetail("database", "Connection failed")
                    .build();
            }
            
            // 2. 필수 외부 의존성 확인 (필요시)
            // Redis, S3 등의 연결 상태 확인
            
            // 3. 애플리케이션 내부 상태 확인
            if (!isApplicationReady()) {
                return Health.down()
                    .withDetail("application", "Not ready")
                    .build();
            }
            
            return Health.up().build();
            
        } catch (Exception e) {
            return Health.down()
                .withDetail("error", e.getMessage())
                .build();
        }
    }
}
```

GitHub Actions 워크플로우에서는 배포 후 다음과 같이 헬스 체크를 수행합니다:

```yaml
- name: Health Check
  run: |
    for i in {1..30}; do
      HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://$INSTANCE_IP:8080/actuator/health)
      if [ $HTTP_STATUS -eq 200 ]; then
        echo "Health check passed"
        exit 0
      fi
      echo "Waiting for application to be ready... ($i/30)"
      sleep 10
    done
    echo "Health check failed after 5 minutes"
    exit 1
```

최대 5분(30회 × 10초) 동안 헬스 체크를 시도하며, 200 OK 응답을 받으면 배포 성공으로 판단합니다.

**자동 롤백 전략:**

헬스 체크가 실패하면 GitHub Actions 워크플로우에서 자동으로 이전 버전으로 롤백합니다:

**1. Docker 이미지 버전 관리:**

배포 시 Docker 이미지를 빌드하면서 두 가지 태그를 생성합니다:
- `latest`: 항상 최신 버전을 가리킴
- `<git-commit-hash>`: 특정 커밋의 버전을 식별

```yaml
- name: Build and Push Docker Image
  run: |
    docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$GITHUB_SHA .
    docker tag $ECR_REGISTRY/$ECR_REPOSITORY:$GITHUB_SHA $ECR_REGISTRY/$ECR_REPOSITORY:latest
    docker push $ECR_REGISTRY/$ECR_REPOSITORY:$GITHUB_SHA
    docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest
```

**2. 롤백 구현:**

배포 실패 시 이전 성공한 커밋의 Docker 이미지로 롤백합니다:

```yaml
- name: Deploy to Instance 1
  id: deploy-1
  run: |
    # 새 버전 배포
    ssh ec2-user@$INSTANCE1_IP "docker pull $ECR_REGISTRY/$ECR_REPOSITORY:$GITHUB_SHA && \
                                 docker stop app || true && \
                                 docker rm app || true && \
                                 docker run -d --name app -p 8080:8080 $ECR_REGISTRY/$ECR_REPOSITORY:$GITHUB_SHA"

- name: Health Check Instance 1
  id: health-check-1
  run: |
    # 헬스 체크 수행 (위의 스크립트)
  continue-on-error: true

- name: Rollback Instance 1
  if: steps.health-check-1.outcome == 'failure'
  run: |
    echo "Deployment failed. Rolling back to previous version..."
    PREVIOUS_COMMIT=$(git rev-parse HEAD~1)
    ssh ec2-user@$INSTANCE1_IP "docker stop app && \
                                 docker rm app && \
                                 docker run -d --name app -p 8080:8080 $ECR_REGISTRY/$ECR_REPOSITORY:$PREVIOUS_COMMIT"
    
    # 롤백 후 헬스 체크
    for i in {1..20}; do
      HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://$INSTANCE1_IP:8080/actuator/health)
      if [ $HTTP_STATUS -eq 200 ]; then
        echo "Rollback successful"
        exit 1  # 배포는 실패로 처리
      fi
      sleep 10
    done
    echo "Rollback health check failed!"
    exit 1
```

**단계적 배포를 통한 리스크 최소화:**

첫 번째 인스턴스 배포 성공을 검증한 후에만 두 번째 인스턴스로 배포를 진행하여, 전체 서비스 장애를 예방했습니다:

```yaml
- name: Deploy to Instance 1
  # ... 배포 로직

- name: Health Check Instance 1
  # ... 헬스 체크

# 인스턴스 1이 성공해야만 인스턴스 2 배포 진행
- name: Deploy to Instance 2
  if: steps.health-check-1.outcome == 'success'
  # ... 배포 로직

- name: Health Check Instance 2
  # ... 헬스 체크
```

이렇게 하면:
1. 인스턴스 1 배포 실패 시: 인스턴스 1만 롤백, 인스턴스 2는 구버전 유지 → 서비스 정상 운영
2. 인스턴스 1 성공, 인스턴스 2 실패 시: 인스턴스 2만 롤백 → 두 인스턴스 모두 신버전으로 통일 실패, 하지만 서비스는 계속 운영

**롤백 실패 시 대응:**

롤백 자체가 실패하는 극단적인 상황에 대비하여 다음과 같은 안전장치를 마련했습니다:

1. **Slack 알림**: 배포 실패 및 롤백 실패 시 즉시 팀원들에게 알림 전송
2. **수동 개입 가이드**: 워크플로우 실패 로그에 수동 롤백 명령어를 출력하여 긴급 대응 가능
3. **이전 버전 이미지 보존**: ECR에서 최소 3개 버전의 이미지를 보관하여 언제든 수동 롤백 가능

```yaml
- name: Notify Deployment Failure
  if: failure()
  run: |
    curl -X POST $SLACK_WEBHOOK_URL \
      -H 'Content-Type: application/json' \
      -d '{
        "text": "🚨 Deployment Failed",
        "blocks": [{
          "type": "section",
          "text": {
            "type": "mrkdwn",
            "text": "*Deployment to production failed*\nCommit: $GITHUB_SHA\nManual rollback command:\n```docker run -d --name app $ECR_REGISTRY/$ECR_REPOSITORY:<previous-commit>```"
          }
        }]
      }'
```

이러한 체계적인 배포 및 롤백 전략을 통해 배포 실패로 인한 서비스 장애를 최소화하고, 문제 발생 시 빠르게 안정적인 상태로 복구할 수 있었습니다.
