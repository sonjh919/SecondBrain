[](슬로우%20쿼리%20로그%20분석%20방법.md)### Question
왜 하나의 쿼리를 두 개로 나누셨고, 각각에 어떤 복합 인덱스를 설계하셨나요? 그리고 어떻게 Covering Index를 충족시킬 수 있었나요?

### Answer
초기 쿼리는 활성화된 토론방으로, 특정 기간 이후에 댓글이 달린 토론방을 댓글과 답글 수, 좋아요 수 등과 함께 토론방 정보를 가져오는 쿼리였습니다. 이때 문제점은 좋아요와 댓글, 답글이 테이블 1:N관계로 카디널리티가 폭증하고 인덱스로는 커버 불가능한 테이블 풀 스캔이 발생하여 속도가 느려지는 것이 원인이었습니다.
그래서 여러 단계를 나누고, 필터링하는 부분을 각각 호출하여 전부 커버링 인덱스를 충족하도록 했습니다. 그리고 마지막에 해당되는 토론방 목록 Id들만 가져와서 한꺼번에 조회하여 속도를 개선했습니다.
활성 토론방 ID 필터링 -> 좋아요 정보 필터링 -> 댓글 수 필터링 -> 토론방 엔티티 필터링 -> 다음 커서용 댓글 ID 필터링을 거쳐 어플리케이션에서 조합하여 속도를 개선했습니다.

### Feedback
문제 원인(1:N 조인으로 인한 카디널리티 폭증)을 정확히 파악하고, 쿼리 분리 전략을 통해 각 단계에서 Covering Index를 적용한 접근이 매우 훌륭합니다. 다만, 각 필터링 단계에서 구체적으로 어떤 복합 인덱스를 설계했는지(예: `(status, created_at, id)` 형태), 그리고 쿼리를 여러 개로 나눴을 때 네트워크 왕복 비용(Round-trip)이 증가하는데도 성능이 개선된 이유를 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
초기 쿼리는 활성화된 토론방 중에서 특정 기간 이후에 댓글이 달린 토론방을 조회하면서, 동시에 댓글/답글 수, 좋아요 수 등의 집계 정보와 토론방 상세 정보를 한 번에 가져오는 복잡한 JOIN 쿼리였습니다.

**문제점 분석:**

이 쿼리의 가장 큰 문제는 토론방(1) : 좋아요(N), 토론방(1) : 댓글(N), 댓글(1) : 답글(N) 관계로 인한 **카디널리티 폭증**이었습니다. 예를 들어 토론방 1개에 좋아요 100개, 댓글 50개, 각 댓글마다 답글 10개가 있다면, JOIN 결과는 1 × 100 × 50 × 10 = 50,000개의 중복 row가 생성됩니다. 이렇게 폭증한 데이터에 대해 GROUP BY와 집계 함수를 적용하면서 성능이 급격히 저하되었습니다.

또한 이렇게 복잡한 JOIN 쿼리에서는 옵티마이저가 효율적인 실행 계획을 수립하기 어렵고, 필요한 모든 컬럼을 하나의 복합 인덱스로 커버하는 것이 불가능했습니다. 결과적으로 테이블 풀 스캔이 발생하여 30초라는 긴 조회 시간이 걸렸습니다.

**해결 전략: 쿼리 분리 + 각각 Covering Index 적용**

복잡한 하나의 쿼리를 독립적인 여러 단계로 분리하여, 각 단계마다 간단하고 명확한 복합 인덱스를 설계하여 Covering Index를 충족시켰습니다:

1. **활성 토론방 ID 필터링**
   - 조건: `status = 'ACTIVE'` AND `created_at >= ?`
   - 복합 인덱스: `(status, created_at, id)`
   - Covering: WHERE 조건과 SELECT 컬럼이 모두 인덱스에 포함

2. **좋아요 정보 필터링**
   - 조건: `discussion_room_id IN (?)` 
   - 복합 인덱스: `(discussion_room_id, id)`
   - Covering: COUNT를 애플리케이션에서 계산하거나, 인덱스만으로 카운트 가능

3. **댓글 수 필터링**
   - 조건: `discussion_room_id IN (?)` AND `created_at >= ?`
   - 복합 인덱스: `(discussion_room_id, created_at, id)`
   - Covering: 조건과 필요한 컬럼이 인덱스에 포함

4. **토론방 엔티티 조회**
   - 조건: `id IN (?)`
   - 기본키 인덱스로 효율적 조회

5. **다음 커서용 댓글 ID 필터링**
   - 페이징을 위한 커서 정보 조회
   - 복합 인덱스로 Covering

각 단계에서는 인덱스만 읽어 필요한 ID 목록을 빠르게 가져오고, 최종적으로 필터링된 토론방 ID들에 대해서만 상세 정보를 조회했습니다. 애플리케이션 레벨에서 각 단계의 결과를 조합하여 최종 결과를 구성했습니다.

**쿼리 분리의 효과:**

여러 번의 쿼리 실행으로 네트워크 왕복(Round-trip)이 증가했지만, 각 쿼리가 모두 Covering Index를 사용하여 인덱스만 읽는 초고속 조회였기 때문에 총 실행 시간은 오히려 크게 감소했습니다. 특히 카디널리티 폭증 문제가 해결되어 불필요한 중복 row 생성과 복잡한 GROUP BY 연산이 제거되었습니다.

결과적으로 30초에서 700ms로 약 97.67%의 성능 개선을 달성했습니다. 이는 복잡한 쿼리를 무조건 하나로 합치는 것보다, 각 단계를 최적화하고 Covering Index를 적용할 수 있도록 분리하는 것이 더 효과적일 수 있다는 것을 보여주는 사례입니다.
