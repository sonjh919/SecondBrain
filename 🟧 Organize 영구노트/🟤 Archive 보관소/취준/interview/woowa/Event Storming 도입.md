### Question
Event Storming을 도입하게 된 구체적인 배경은 무엇이고, 어떻게 진행하셨나요?

### Answer
프로젝트 초기에 각자 다른 비즈니스 흐름을 맡아 진행하다 보니, 각자 이해하는 흐름이 달라서 기획에서 충돌하는 경우가 있었습니다. 그래서 Event Storming을 통해 팀원들이 모여서 도메인 이벤트를 시간 순서대로 포스트잇에 붙이면서 비즈니스 플로우를 시각화했습니다.
이 과정에서 팀원들의 도메인 이해가 통일되어 이후 개발 단계에서 불필요한 커뮤니케이션 비용을 크게 줄일 수 있었습니다.

### Feedback
문제 상황과 해결 방법, 그리고 개선 효과를 명확하게 설명하셨습니다. 다만, Event Storming의 구체적인 진행 단계(도메인 이벤트 식별 → 커맨드 식별 → 애그리게잇 도출 등), 어떤 색상의 포스트잇을 어떤 용도로 사용했는지, 그리고 Event Storming 결과가 실제 설계나 코드에 어떻게 반영되었는지 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다.

### Recommendation
**Event Storming 도입 배경:**

이선좌 프로젝트는 티켓팅 시스템으로, 공연 등록, 티켓 예매, 결제, 좌석 배정 등 여러 비즈니스 흐름이 복잡하게 얽혀 있는 도메인이었습니다.

프로젝트 초기에 팀원들이 각자 다른 기능을 맡아 개발을 시작했는데, 다음과 같은 문제가 발생했습니다:

1. **도메인 이해 차이**: 
   - A 팀원: "티켓 예매 시 즉시 좌석이 확정된다"
   - B 팀원: "티켓 예매 후 결제 완료 시 좌석이 확정된다"
   - 이러한 이해 차이로 인해 API 설계와 데이터베이스 스키마가 서로 맞지 않는 상황 발생

2. **기획 충돌**:
   - 각자 생각한 비즈니스 플로우가 달라서 회의 시 "이게 맞다/아니다"로 의견이 충돌
   - 요구사항 해석의 차이로 인해 같은 기능을 다르게 구현

3. **커뮤니케이션 비용 증가**:
   - 기능 간 연동 시점에 "이 필드가 왜 있어요?", "이 상태값은 언제 바뀌나요?" 등의 질문이 반복
   - 개발 중간에 설계를 다시 논의하고 수정하는 일이 빈번

이러한 문제를 해결하기 위해 **모든 팀원이 도메인에 대한 공통된 이해를 가질 필요**가 있었고, DDD의 Event Storming 기법을 도입하기로 결정했습니다.

---

**Event Storming 진행 과정:**

Event Storming은 Alberto Brandolini가 고안한 워크샵 형태의 협업 기법으로, 도메인 전문가와 개발자가 함께 도메인을 탐색하고 이해하는 과정입니다.

**준비물:**
- 큰 벽면 또는 화이트보드
- 여러 색상의 포스트잇 (각 색상은 특정 의미를 가짐)
- 마커 펜
- 팀원 전체 참여

**1단계: 도메인 이벤트 식별 (주황색 포스트잇)**

팀원들이 모여서 시스템에서 발생하는 **도메인 이벤트**를 브레인스토밍했습니다. 도메인 이벤트는 "~되었다", "~이 발생했다" 형식의 과거형 문장으로 표현합니다.

```
[주황색] 공연이 등록되었다
[주황색] 티켓이 예매되었다
[주황색] 결제가 완료되었다
[주황색] 좌석이 배정되었다
[주황색] 예매가 취소되었다
```

이벤트를 **시간 순서대로** 벽에 붙이면서 비즈니스 플로우를 시각화했습니다:

```
[공연이 등록되었다] → [티켓이 예매되었다] → [결제가 완료되었다] → [좌석이 배정되었다]
```

이 과정에서:
- "티켓 예매 후 바로 좌석이 배정되나요?"
- "결제 전에도 좌석을 확인할 수 있나요?"

등의 질문이 나오면서 팀원들의 이해가 서로 다르다는 것을 발견했고, 논의를 통해 정확한 플로우를 합의했습니다.

**2단계: 커맨드 식별 (파란색 포스트잇)**

각 도메인 이벤트를 발생시키는 **커맨드(명령)**를 식별했습니다. 커맨드는 "~하다", "~을 생성하다" 형식의 명령형 문장입니다.

```
[파란색] 공연 등록하다 → [주황색] 공연이 등록되었다
[파란색] 티켓 예매하다 → [주황색] 티켓이 예매되었다
[파란색] 결제하다 → [주황색] 결제가 완료되었다
```

**3단계: 액터 식별 (노란색 포스트잇)**

커맨드를 실행하는 **액터(사용자 또는 시스템)**를 식별했습니다.

```
[노란색] 관리자 → [파란색] 공연 등록하다
[노란색] 일반 사용자 → [파란색] 티켓 예매하다
[노란색] 결제 시스템 → [파란색] 결제하다
```

**4단계: 애그리게잇 도출 (연한 노란색 큰 포스트잇)**

관련된 이벤트와 커맨드를 묶어서 **애그리게잇(Aggregate)**을 도출했습니다. 애그리게잇은 일관성을 유지해야 하는 도메인 객체들의 묶음입니다.

```
┌─────────────────────────┐
│   공연 애그리게잇         │
│ - 공연 등록하다          │
│ - 공연이 등록되었다       │
│ - 공연 정보 수정하다      │
└─────────────────────────┘

┌─────────────────────────┐
│   예매 애그리게잇         │
│ - 티켓 예매하다          │
│ - 티켓이 예매되었다       │
│ - 예매 취소하다          │
└─────────────────────────┘
```

**5단계: 바운디드 컨텍스트 정의**

애그리게잇들을 분석하여 **바운디드 컨텍스트(Bounded Context)**를 도출했습니다:

- **공연 관리 컨텍스트**: 공연 등록, 수정, 조회
- **예매 컨텍스트**: 티켓 예매, 취소
- **결제 컨텍스트**: 결제 처리
- **좌석 관리 컨텍스트**: 좌석 배정, 반환

---

**Event Storming 결과의 실제 반영:**

Event Storming을 통해 도출한 결과를 실제 설계와 코드에 다음과 같이 반영했습니다:

**1. 도메인 모델 설계:**

```java
// 공연 애그리게잇
@Entity
public class Performance {
    @Id
    private Long id;
    private String title;
    private PerformanceStatus status;
    
    // 도메인 이벤트: 공연이 등록되었다
    public void register() {
        this.status = PerformanceStatus.REGISTERED;
        // 이벤트 발행
    }
}

// 예매 애그리게잇
@Entity
public class Reservation {
    @Id
    private Long id;
    private ReservationStatus status;
    
    // 도메인 이벤트: 티켓이 예매되었다
    public void reserve() {
        this.status = ReservationStatus.RESERVED;
    }
    
    // 도메인 이벤트: 결제가 완료되었다 → 좌석이 배정되었다
    public void completePayment() {
        this.status = ReservationStatus.CONFIRMED;
    }
}
```

**2. 바운디드 컨텍스트별 모듈 구성:**

Multi-Module 구조를 설계할 때 Event Storming에서 도출한 바운디드 컨텍스트를 참고했습니다:

```
domain-module/
├── performance/  (공연 관리 컨텍스트)
├── reservation/  (예매 컨텍스트)
├── payment/      (결제 컨텍스트)
└── seat/         (좌석 관리 컨텍스트)
```

**3. API 설계:**

Event Storming의 커맨드와 이벤트를 기반으로 RESTful API를 설계했습니다:

```
POST /api/performances        # 공연 등록하다
POST /api/reservations        # 티켓 예매하다
POST /api/payments            # 결제하다
DELETE /api/reservations/{id} # 예매 취소하다
```

---

**Event Storming의 효과:**

1. **도메인 이해 통일**: 
   - 모든 팀원이 동일한 비즈니스 플로우를 이해하게 됨
   - "티켓 예매와 좌석 배정의 타이밍"과 같은 핵심 이슈가 명확해짐

2. **커뮤니케이션 비용 감소**:
   - 개발 중 "이게 맞나요?" 식의 질문이 80% 이상 감소
   - 명확한 도메인 용어(유비쿼터스 언어)가 정립되어 의사소통이 원활해짐

3. **설계 품질 향상**:
   - 애그리게잇과 바운디드 컨텍스트가 명확해져 응집도 높은 설계 가능
   - 불필요한 의존성을 사전에 방지

4. **개발 속도 향상**:
   - 초기에 시간을 투자했지만, 개발 단계에서 재작업이 거의 없어 전체적으로 빨라짐
   - 기능 간 연동 시 인터페이스가 명확하여 통합이 수월함

Event Storming은 단순히 다이어그램을 그리는 것이 아니라, **팀원들이 함께 도메인을 탐색하고 이해를 공유하는 협업의 장**이었습니다. 이를 통해 프로젝트 초기에 발생할 수 있는 많은 문제를 사전에 예방할 수 있었습니다.
