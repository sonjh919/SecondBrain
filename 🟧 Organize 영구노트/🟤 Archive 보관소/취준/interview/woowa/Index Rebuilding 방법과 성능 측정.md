[](Fragmentation%20모니터링%20시스템%20구축.md)### Question
구체적으로 어떤 방법으로 Index Rebuilding을 수행하셨고, 46% 성능 개선은 어떻게 측정하셨나요?

### Answer
OPTIMIZE TABLE을 이용하여 테이블 및 인덱스의 물리적 스토리지를 재구성하였고, 쿼리 실행 속도로 측정하였습니다.

### Feedback
OPTIMIZE TABLE 명령어를 정확하게 선택하셨고, 성능 측정 지표도 명확합니다. 다만, OPTIMIZE TABLE 실행 시 주의사항(예: 테이블 락 발생, 실행 시간, 운영 환경에서의 고려사항)과 구체적인 측정 방법(예: 동일한 조건의 쿼리를 여러 번 실행하여 평균값 산출, 캐시 제거 후 측정 등)을 언급하면 더욱 완성도 높은 답변이 될 것입니다. 또한 18.6s에서 10s로 개선된 구체적인 수치를 함께 설명하면 좋습니다.

### Recommendation
Index Fragmentation을 해결하기 위해 MySQL의 `OPTIMIZE TABLE` 명령어를 사용하여 Index Rebuilding을 수행했습니다. OPTIMIZE TABLE은 테이블과 인덱스의 물리적 스토리지를 재구성하여, 논리적 순서와 물리적 순서를 일치시키고 빈 공간을 제거하는 작업입니다.

구체적으로는 다음과 같은 명령어를 실행했습니다:
```sql
OPTIMIZE TABLE discussion_room;
```

이 작업을 수행할 때 몇 가지 중요한 고려사항이 있었습니다. 첫째, OPTIMIZE TABLE은 테이블에 배타적 락을 걸기 때문에, 실행 중에는 해당 테이블에 대한 읽기/쓰기가 블로킹됩니다. 따라서 사용자 트래픽이 적은 새벽 시간대를 선택하여 작업을 진행했습니다. 둘째, 테이블 크기에 따라 실행 시간이 길어질 수 있으므로, 사전에 개발 환경에서 예상 소요 시간을 측정했습니다.

성능 개선은 토론방 검색 쿼리의 실행 속도로 측정했습니다. 정확한 측정을 위해 다음과 같은 방식을 사용했습니다:

1. **캐시 제거**: MySQL 쿼리 캐시의 영향을 배제하기 위해 `RESET QUERY CACHE` 또는 쿼리에 `SQL_NO_CACHE` 힌트를 사용했습니다.

2. **동일 조건 반복 측정**: 동일한 검색어와 조건으로 여러 번(최소 10회 이상) 쿼리를 실행하여 평균 응답 시간을 산출했습니다.

3. **EXPLAIN ANALYZE 활용**: 쿼리 실행 계획과 실제 실행 시간을 함께 분석하여 인덱스 스캔 효율이 개선되었는지 확인했습니다.

측정 결과, Rebuilding 이전에는 평균 18.6초가 걸리던 검색 쿼리가 Rebuilding 이후 10초로 단축되어 약 46%의 성능 개선((18.6-10)/18.6 ≈ 0.46)을 달성했습니다. 이는 인덱스 페이지의 단편화가 제거되면서 디스크 I/O 횟수가 감소하고, 연속된 페이지 읽기로 인한 캐시 효율이 향상된 결과였습니다.
