[](인덱스와%20클러스터링-논클러스터링%20차이.md)### Question
어떤 상황에서 Unique 제약조건을 추가하게 되었고, 이것이 어떻게 성능 개선으로 이어졌나요?

### Answer
회원가입 시 닉네임 중복 검사 쿼리에서 실행 계획을 확인했더니, nickname 필드에 인덱스가 없어 테이블 풀스캔이 발생하고 있었습니다.
그래서 닉네임에 UNIQUE 제약조건을 추가하고 논클러스터링 인덱스를 생성하여 성능 개선을 이루어냈습니다.

### Feedback
문제 상황과 해결 방법을 명확하게 설명하셨습니다. 다만, UNIQUE 제약조건을 추가할 때 자동으로 논클러스터링 인덱스가 생성되는 점, 그리고 UNIQUE 인덱스가 일반 인덱스보다 더 효율적인 이유(조회 시 첫 번째 매칭 발견 후 즉시 중단 가능)를 추가로 설명하면 더욱 완성도 높은 답변이 될 것입니다. 또한 230ms에서 0.2ms로 개선된 구체적인 수치를 언급하면 좋습니다.

### Recommendation
회원가입 플로우에서 닉네임 중복 검사 API가 병목 지점으로 식별되었습니다. 슬로우 쿼리 로그를 분석한 결과, 닉네임 중복 확인 쿼리가 평균 230ms가 소요되고 있었습니다.

**문제 원인 분석:**

EXPLAIN 명령어로 쿼리 실행 계획을 확인한 결과, 다음과 같은 문제를 발견했습니다:

```sql
EXPLAIN SELECT id FROM users WHERE nickname = 'testuser';
```

실행 계획에서 `type: ALL`이 표시되어 테이블 풀 스캔이 발생하고 있었습니다. nickname 필드에 인덱스가 존재하지 않아, 중복 검사를 위해 users 테이블의 모든 row를 스캔해야 했고, 사용자가 증가할수록 성능이 선형적으로 저하되는 구조였습니다.

**해결 방법: UNIQUE 제약조건 + 논클러스터링 인덱스**

닉네임은 비즈니스 요구사항상 시스템 내에서 고유해야 하므로, UNIQUE 제약조건을 추가하는 것이 자연스러웠습니다:

```sql
ALTER TABLE users ADD CONSTRAINT uk_nickname UNIQUE (nickname);
```

MySQL(InnoDB)에서는 UNIQUE 제약조건을 추가하면 자동으로 해당 컬럼에 **논클러스터링 인덱스(Non-Clustered Index)**가 생성됩니다. 이는 Primary Key와 달리 데이터의 물리적 순서를 변경하지 않으면서, 별도의 인덱스 구조(B-Tree)를 생성하여 빠른 검색을 가능하게 합니다.

**성능 개선 효과:**

UNIQUE 제약조건 추가 후, 동일한 쿼리의 실행 계획을 다시 확인했습니다:

```sql
EXPLAIN SELECT id FROM users WHERE nickname = 'testuser';
```

이번에는 `type: const` 또는 `type: ref`가 표시되며, `key: uk_nickname`으로 인덱스를 사용하고 있음을 확인했습니다. 또한 `rows: 1`로 단 1개의 row만 검사하면 됨을 알 수 있었습니다.

UNIQUE 인덱스는 일반 인덱스보다 더 효율적인데, 이는 다음과 같은 이유 때문입니다:

1. **Early Termination**: UNIQUE 제약조건으로 인해 중복이 불가능하므로, 옵티마이저는 첫 번째 매칭을 발견하는 즉시 검색을 중단할 수 있습니다.

2. **최적화된 실행 계획**: 데이터베이스 옵티마이저가 UNIQUE 특성을 활용하여 보다 효율적인 실행 계획을 수립합니다.

**측정 결과:**

인덱스 추가 전후로 동일한 닉네임 중복 검사 쿼리를 100회 반복 실행하여 평균 응답 시간을 측정한 결과:

- **Before**: 평균 230ms (테이블 풀 스캔)
- **After**: 평균 0.2ms (UNIQUE 인덱스 활용)
- **개선율**: (230 - 0.2) / 230 ≈ **99.91% 개선**

이러한 극적인 성능 개선은 O(n)의 선형 검색이 O(log n)의 B-Tree 검색으로 전환되었기 때문입니다. 특히 사용자 수가 증가하더라도 로그 시간 복잡도 덕분에 성능이 안정적으로 유지될 수 있게 되었습니다.

**추가 효과:**

성능 개선 외에도 다음과 같은 부가적인 이점이 있었습니다:

1. **데이터 무결성 보장**: DB 레벨에서 닉네임 중복을 방지하여, 동시성 이슈(Race Condition)로 인한 중복 발생을 원천 차단했습니다.

2. **애플리케이션 로직 간소화**: 중복 검사 후 삽입하는 복잡한 로직 대신, 단순 INSERT 시도로 중복 여부를 판단할 수 있게 되었습니다(예외 처리 활용).
