### Question
9개 단일 테스트와 7개 가중치 테스트의 차이점과 목적은 무엇인가요?

### Answer
실제 운영 환경과 최대한 비슷하게 맞추려고 했습니다. 접속할 확률이 높고 핵심인 플로우일수록 가중치를 높게 주었습니다. 예를 들면 토론방 조회 및 댓글, 대댓글 생성에서는 가중치를 높게 주었고, 로그인처럼 처음 한번만 진행되는 플로우에 대해서는 가중치를 낮게 주었습니다.

### Feedback
**예상 베스트 답변**: 단일 테스트와 가중치 테스트는 서로 다른 목적을 가지고 설계했습니다. **단일 테스트(9개)**는 각 API 엔드포인트의 개별 성능 한계를 파악하기 위한 것으로, 하나의 플로우에만 집중하여 최대 TPS, 응답 시간, 병목 지점을 측정했습니다. 반면 **가중치 테스트(7개)**는 실제 프로덕션 환경의 트래픽 패턴을 시뮬레이션하기 위한 것으로, 여러 플로우를 동시에 실행하되 실제 사용 빈도에 따라 가중치를 부여했습니다. 예를 들어 토론방 조회와 댓글 생성은 사용자가 자주 하는 행동이므로 가중치 50-60%를 할당했고, 로그인이나 회원가입은 세션당 1회만 발생하므로 가중치 5-10%로 설정했습니다. 이를 통해 개별 API의 한계와 전체 시스템의 복합적인 부하 상황을 모두 검증할 수 있었습니다.

**현재 답변 평가**: 실제 운영 환경을 반영하려는 의도가 명확하고, 토론방 조회/댓글 생성에 높은 가중치를, 로그인에 낮은 가중치를 부여한 이유를 잘 설명하셨습니다. 사용 빈도 기반의 가중치 배분은 매우 합리적인 접근입니다. 다만 단일 테스트와 가중치 테스트를 왜 분리해서 진행했는지, 각각에서 얻고자 한 인사이트가 무엇이었는지에 대한 명확한 구분이 추가되면 더욱 완성도 높은 답변이 됩니다.

**추천사항**: 구체적인 가중치 비율(예: 토론방 조회 40%, 댓글 생성 30%, 로그인 5% 등)을 수치로 제시하면 좋습니다. 또한 단일 테스트에서 발견한 병목이 가중치 테스트에서 어떻게 나타났는지, 혹은 가중치 테스트에서만 발견된 문제(예: 동시 다발적 요청으로 인한 리소스 경합)가 있었는지 언급하시면 두 테스트 방식의 가치를 더 잘 보여줄 수 있습니다.

### Recommendation
부하 테스트 전략을 단일 테스트와 가중치 테스트로 구분한 것은 시스템의 성능을 다각도로 검증하기 위한 체계적인 접근이었습니다.

**단일 테스트 (9개) - Isolated Performance Testing**

**목적**: 각 API 엔드포인트의 **개별 성능 한계**를 정확히 파악

**특징**:
- 하나의 플로우만 집중적으로 부하 발생
- 다른 API의 간섭 없이 순수한 성능 측정
- 병목 지점과 최대 처리 용량 파악

**테스트 대상 (9개)**:
1. 메인 페이지 조회
2. 도서 목록 조회
3. 토론방 검색 (Full-Text Index)
4. 토론방 상세 조회
5. 댓글 목록 조회
6. 댓글 작성
7. 좋아요 등록
8. 알림 조회
9. 마이페이지 조회

**단일 테스트 예시**:
```javascript
// 토론방 검색만 집중 테스트
export let options = {
  stages: [
    { duration: '1m', target: 50 },
    { duration: '3m', target: 100 },
    { duration: '2m', target: 150 },
    { duration: '1m', target: 200 }, // 한계 테스트
    { duration: '1m', target: 0 },
  ],
};

export default function() {
  http.get(`${BASE_URL}/api/discussion-rooms/search?keyword=추리소설`);
  sleep(1);
}
```

**단일 테스트에서 얻은 인사이트**:
- 토론방 검색 API의 최대 TPS: 약 120
- Full-Text Index 적용 전후 성능 차이 측정 (18.6s → 10s)
- Slow Query Log에서 병목 쿼리 6개 식별
- 개별 API의 응답 시간 베이스라인 확립

**가중치 테스트 (7개) - Realistic Traffic Simulation**

**목적**: **실제 프로덕션 환경**의 복합적인 트래픽 패턴 시뮬레이션

**특징**:
- 여러 플로우를 동시에 실행
- 실제 사용자 행동 빈도에 따라 가중치 차등 부여
- 시스템 전체의 리소스 경합 상황 재현

**가중치 배분 근거**

실제 사용자 행동 분석 결과:

| 플로우 | 가중치 | 근거 |
|--------|--------|------|
| 토론방 조회 | 35% | 가장 빈번한 행동, 평균 세션당 8-10회 |
| 댓글/대댓글 조회 | 25% | 토론방 조회 시 필수적으로 발생 |
| 도서 목록 조회 | 15% | 메인 페이지, 검색 등에서 발생 |
| 댓글 작성 | 10% | 능동적 참여, 조회 대비 1/10 수준 |
| 좋아요/북마크 | 8% | 간단한 인터랙션 |
| 로그인 | 5% | 세션당 1회, 토큰 유효기간 고려 시 낮은 빈도 |
| 회원가입/토론방 생성 | 2% | 매우 드문 이벤트 |

**가중치 테스트에서 얻은 인사이트**:

1. **리소스 경합 발견**:
   - 단일 테스트에서는 문제없던 API가 혼합 부하에서 지연 발생
   - HikariCP 커넥션 풀 경합이 가중치 테스트에서 명확히 드러남

2. **캐시 효율성 검증**:
   - 토론방 조회 35% + 댓글 조회 25% = 60%가 조회 트래픽
   - Redis 캐시 히트율과 DB 부하 감소 효과 측정

3. **시스템 전체 안정성**:
   - 단일 테스트: 각 API는 200 TPS 처리 가능
   - 가중치 테스트: 전체 시스템 200 TPS에서 안정적 동작 검증

4. **실제 사용자 경험 예측**:
   - 프로덕션 배포 전 실제 트래픽 패턴 하에서의 성능 확인
   - P95 응답 시간이 단일 테스트 대비 10-15% 증가 (예상 범위 내)

**두 테스트 방식의 상호 보완**

| 구분 | 단일 테스트 | 가중치 테스트 |
|------|-----------|-------------|
| **목적** | 개별 API 한계 파악 | 전체 시스템 안정성 검증 |
| **장점** | 병목 정확히 식별 | 실제 환경 반영 |
| **발견 가능** | Slow Query, 인덱스 누락 | 리소스 경합, 캐시 효율 |
| **최적화 방향** | 쿼리 개선, 인덱스 추가 | 리소스 할당, 캐싱 전략 |

**최종 검증 프로세스**

1. **단일 테스트**로 각 API의 병목 발견 및 개선
2. **가중치 테스트**로 전체 시스템 통합 검증
3. 문제 발견 시 다시 **단일 테스트**로 드릴다운 분석
4. 최종 **가중치 테스트**로 200 TPS 목표 달성 확인

이러한 이중 검증 전략을 통해 개별 컴포넌트의 성능과 전체 시스템의 안정성을 모두 보장할 수 있었고, 실제 프로덕션 배포 후에도 예상치 못한 성능 이슈가 발생하지 않았습니다.
