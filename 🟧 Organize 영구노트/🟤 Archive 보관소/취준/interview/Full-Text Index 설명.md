[](Index%20Fragmentation%20모니터링.md)### Question
Full-Text Index에 대해서 자세하게 설명해줄 수 있나요?

### Answer
일반 인덱스와는 다르게 문자열을 나눠서 토큰화한다음 토큰을 인덱스로 설정하고, 역인덱스 방식으로 동작하기 때문에 일반적인 like검색보다 빠릅니다.

### Feedback
**예상 베스트 답변**: Full-Text Index는 텍스트 데이터를 형태소 단위로 분리하여 토큰화하고, 각 토큰에 대해 역인덱스(Inverted Index) 구조를 생성합니다. 역인덱스는 단어를 키로, 해당 단어가 포함된 문서 ID를 값으로 저장하여 검색 시 O(1)에 가까운 속도로 접근할 수 있습니다. 일반 B-Tree 인덱스는 LIKE '%keyword%' 같은 중간 일치 검색에서 Full Scan이 발생하지만, Full-Text Index는 토큰 기반으로 즉시 매칭되는 문서를 찾을 수 있어 대용량 텍스트 검색에 효과적입니다.

**현재 답변 평가**: 핵심 개념인 토큰화와 역인덱스를 정확히 언급하셨고, 일반 인덱스와의 차이점도 잘 설명하셨습니다. 다만 역인덱스가 구체적으로 어떤 구조로 데이터를 저장하는지, 그리고 실제 프로젝트에서 어떤 검색 쿼리에 적용했는지에 대한 구체적인 예시가 추가되면 더욱 설득력 있는 답변이 될 것입니다.

**추천사항**: MySQL의 Full-Text Index 파서 종류(ngram, mecab 등), 최소 검색 단어 길이(ft_min_word_len), Boolean Mode vs Natural Language Mode 같은 실무적 설정 경험을 언급하시면 기술 깊이를 더 보여줄 수 있습니다.

### Recommendation
Full-Text Index는 텍스트 검색을 위한 특수한 인덱스 구조로, 일반 B-Tree 인덱스와는 근본적으로 다른 방식으로 동작합니다. 

먼저 **토큰화 과정**을 통해 문자열을 의미 있는 단위로 분리합니다. MySQL에서는 ngram 파서나 mecab 같은 형태소 분석기를 사용할 수 있으며, 각 언어의 특성에 맞게 토큰을 생성합니다. 예를 들어 "독서 토론 서비스"라는 문자열은 "독서", "토론", "서비스"로 분리됩니다.

다음으로 **역인덱스(Inverted Index) 구조**를 생성합니다. 이는 단어를 키로, 해당 단어가 포함된 문서 ID 리스트를 값으로 저장하는 방식입니다. 예를 들어:
- "독서" → [문서1, 문서3, 문서5]
- "토론" → [문서1, 문서2, 문서4]

이러한 구조 덕분에 검색 시 토큰에 대한 문서를 O(1)에 가까운 시간으로 찾을 수 있습니다.

**일반 인덱스와의 차이점**은 명확합니다. B-Tree 인덱스는 `LIKE '%키워드%'` 같은 중간 일치 검색에서 인덱스를 활용할 수 없어 Full Table Scan이 발생하지만, Full-Text Index는 토큰 기반으로 즉시 매칭되는 문서를 찾을 수 있습니다. 

TodokTodok 프로젝트에서는 도서 제목이나 토론 내용 검색에 Full-Text Index를 적용하여 18.6초에서 10초로 46%의 성능 개선을 달성했습니다. 특히 MySQL의 **MATCH() AGAINST()** 문법을 사용했으며, Boolean Mode를 활용하여 AND/OR 조건 검색도 지원했습니다. 또한 ft_min_word_len을 2로 설정하여 한글 2음절 이상 단어부터 검색 가능하도록 최적화했습니다.
