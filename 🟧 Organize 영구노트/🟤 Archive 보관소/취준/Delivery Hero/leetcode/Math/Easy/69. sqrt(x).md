#Math #BinarySearch

## 풀이 코드
```java
class Solution {
    public int mySqrt(int x) {
        // 1. 0과 1은 제곱근이 자기 자신
        // why: 특수한 경우를 먼저 처리하여 이진 탐색 단순화
        if (x < 2) {
            return x;
        }
        
        // 2. 이진 탐색 범위 설정
        // why: 제곱근은 항상 2와 x/2 사이에 존재 (x >= 2일 때)
        int left = 2;
        int right = x / 2;
        
        // 3. 이진 탐색으로 제곱근 찾기
        // why: 정렬된 범위에서 O(log n) 시간에 탐색
        while (left <= right) {
            int mid = left + (right - left) / 2;
            
            // 4. 오버플로우 방지를 위해 나눗셈으로 비교
            // why: mid * mid는 int 범위를 초과할 수 있음
            long num = (long) mid * mid;
            
            // 5. 정확한 제곱근을 찾으면 반환
            // why: mid^2 == x인 경우
            if (num == x) {
                return mid;
            }
            // 6. mid^2가 x보다 크면 왼쪽 절반 탐색
            // why: 더 작은 값을 찾아야 함
            else if (num > x) {
                right = mid - 1;
            }
            // 7. mid^2가 x보다 작으면 오른쪽 절반 탐색
            // why: 더 큰 값을 찾되, mid는 가능한 답이므로 left 이동
            else {
                left = mid + 1;
            }
        }
        
        // 8. 반올림 내림한 제곱근 반환
        // why: left > right가 되면 right가 내림한 제곱근
        return right;
    }
}
```

**시간 복잡도**: O(log x) - 이진 탐색으로 범위를 절반씩 줄임  
**공간 복잡도**: O(1) - 포인터 변수만 사용

## 면접 대본 (영어)

"Let me explain my approach to computing the square root without using built-in functions.

The key insight is to use binary search. We're looking for the largest integer whose square is less than or equal to x. Since squares increase monotonically with the base number, we can use binary search on the range of possible answers.

First, I handle edge cases: if x is 0 or 1, the square root is x itself, so I return it immediately.

For the search range, I set left to 2 and right to x/2. Why x/2? Because for any x >= 2, the square root is always less than or equal to x/2. For example, sqrt(4) = 2 = 4/2, and sqrt(8) ≈ 2.83 < 4 = 8/2. This optimization reduces our search space.

Here's how the algorithm works: I perform standard binary search. At each iteration, I calculate the middle value. Then I need to check if mid squared equals, is less than, or is greater than x. However, I'm careful about overflow - mid * mid could exceed the integer range. So I use a long to store the multiplication result.

If mid squared equals x, I've found the exact square root, so I return mid immediately. If mid squared is greater than x, the answer must be smaller, so I search the left half by setting right to mid - 1. If mid squared is less than x, the answer could be mid or something larger, so I search the right half by setting left to mid + 1.

When the loop exits with left > right, right points to the largest integer whose square doesn't exceed x, which is exactly the floor of the square root we're looking for.

Let me walk through an example: x = 8. Left = 2, right = 4. Mid = 3, 3² = 9 > 8, so right = 2. Now left = 2, right = 2, mid = 2, 2² = 4 < 8, so left = 3. Now left > right, we exit and return right = 2. This is correct: sqrt(8) ≈ 2.83, and we return 2.

Another example: x = 4. Left = 2, right = 2, mid = 2, 2² = 4 == 4, return 2 immediately. Perfect square case.

A larger example: x = 16. Left = 2, right = 8, mid = 5, 5² = 25 > 16, right = 4. Mid = 3, 3² = 9 < 16, left = 4. Mid = 4, 4² = 16 == 16, return 4.

An interesting case: x = 2. Left = 2, right = 1. But wait, left > right initially! Actually, x/2 = 1, so right = 1, left = 2, and left > right from the start. We return right = 1, which is correct: sqrt(2) ≈ 1.41, floor is 1.

Why does returning right work when left > right? During binary search, right always points to a value we know is not too large - its square is ≤ x (or we haven't checked yet). When left overtakes right, left has moved to a value whose square is too large, while right is at the last value whose square was acceptable.

The time complexity is O(log x) because we're doing binary search on a range from 2 to x/2. The space complexity is O(1) since we only use a few variables.

This problem demonstrates several important concepts: binary search on answer space, overflow prevention, and understanding when to use long for intermediate calculations. The binary search approach is much more efficient than trying every integer sequentially, which would be O(√x), and it's the standard method when you can't use built-in math functions."
