#Math

## 풀이 코드
```java
class Solution {
    public boolean isPalindrome(int x) {
        // 1. 음수이거나 10의 배수(0 제외)는 팰린드롬이 아님
        // why: 음수는 -부호 때문에, 10의 배수는 끝자리 0 때문에 대칭 불가
        if (x < 0 || (x % 10 == 0 && x != 0)) {
            return false;
        }
        
        // 2. 숫자의 절반을 뒤집어서 비교
        // why: 전체를 뒤집으면 오버플로우 위험, 절반만 뒤집으면 안전
        int reversed = 0;
        
        // 3. 원본의 절반을 처리할 때까지 반복
        // why: x가 reversed보다 작거나 같아지면 절반 이상 처리한 것
        while (x > reversed) {
            // 4. 뒤집은 숫자를 한 자리씩 구성
            // why: x의 마지막 자리를 떼어 reversed의 끝에 추가
            reversed = reversed * 10 + x % 10;
            x /= 10;
        }
        
        // 5. 짝수 길이는 x == reversed, 홀수 길이는 x == reversed / 10
        // why: 홀수 길이는 중간 숫자가 reversed에 포함되므로 제거 후 비교
        return x == reversed || x == reversed / 10;
    }
}
```

**시간 복잡도**: O(log n) - 숫자의 자릿수만큼 반복 (n은 입력 숫자)  
**공간 복잡도**: O(1) - reversed 변수만 사용

## 면접 대본 (영어)

"Let me explain my approach to checking if a number is a palindrome without converting it to a string.

The key insight is that we don't need to reverse the entire number - we only need to reverse half of it and compare. This avoids potential integer overflow issues and is more efficient. The challenge is determining when we've processed half the digits.

I'll handle edge cases first, then use a mathematical approach to reverse half the number.

First, I check for cases that can't be palindromes: negative numbers are never palindromes because of the minus sign, and numbers ending in zero (except zero itself) can't be palindromes because they would need to start with zero, which doesn't make sense for a regular integer representation.

Here's how the algorithm works: I initialize reversed to 0. Then I enter a loop that continues while x is greater than reversed. In each iteration, I extract the last digit of x using the modulo operation, add it to reversed (after multiplying reversed by 10 to shift existing digits left), and remove the last digit from x by dividing by 10.

The clever part is knowing when to stop: when x becomes less than or equal to reversed, we've processed at least half the digits. For even-length numbers, x will equal reversed. For odd-length numbers, x will equal reversed divided by 10, because reversed will have one extra digit in the middle.

Let me walk through an example: x = 12321. Initially reversed = 0. First iteration: reversed = 0*10 + 1 = 1, x = 1232. Second: reversed = 1*10 + 2 = 12, x = 123. Third: reversed = 12*10 + 3 = 123, x = 12. Now x < reversed, we stop. We check: is x == reversed? No, 12 ≠ 123. Is x == reversed/10? Yes, 12 == 123/10 = 12. So it's a palindrome.

Another example: x = 121. Reversed = 0. First: reversed = 1, x = 12. Second: reversed = 12, x = 1. Now x < reversed, stop. Is x == reversed? No. Is x == reversed/10? Yes, 1 == 12/10 = 1. Palindrome confirmed.

A failing case: x = 10. We immediately return false because x % 10 == 0 and x != 0.

Another failing case: x = 1234. Reversed = 0. First: reversed = 4, x = 123. Second: reversed = 43, x = 12. Third: reversed = 432, x = 1. Now x < reversed. Is 1 == 432? No. Is 1 == 432/10 = 43? No. Not a palindrome.

Why does checking both conditions work? For even-length palindromes like 1221, when we finish, we have x = 12 and reversed = 12. They're equal. For odd-length palindromes like 12321, we have x = 12 and reversed = 123. The middle digit doesn't matter for palindrome property, so we compare x with reversed/10.

The time complexity is O(log₁₀(n)) where n is the value of x, because we're essentially processing each digit once, and the number of digits is logarithmic in the value. The space complexity is O(1) since we only use a constant amount of extra space.

This solution elegantly satisfies the follow-up requirement of not converting to a string. It's also more efficient than string conversion because we avoid the overhead of string operations and only process half the digits. The mathematical approach showcases understanding of number manipulation and clever use of the division operation to detect when we've reached the middle."
