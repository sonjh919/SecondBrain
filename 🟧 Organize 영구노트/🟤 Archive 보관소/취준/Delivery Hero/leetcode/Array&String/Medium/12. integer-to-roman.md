## 풀이 코드
```java
class Solution {
    public String intToRoman(int num) {
        // 1. 값과 심볼을 큰 것부터 작은 것 순서로 배열에 저장
        // why: 큰 단위부터 차감하며 변환하기 위함 (그리디)
        int[] values = {1000, 900, 500, 400, 100, 90, 50, 40, 10, 9, 5, 4, 1};
        String[] symbols = {"M", "CM", "D", "CD", "C", "XC", "L", "XL", "X", "IX", "V", "IV", "I"};
        
        // 2. 결과 문자열을 저장할 StringBuilder
        // why: 문자열 연결을 효율적으로 처리
        StringBuilder result = new StringBuilder();
        
        // 3. 값 배열을 순회하며 로마 숫자 생성
        // why: 각 단위별로 최대한 사용 가능한 만큼 추가
        for (int i = 0; i < values.length; i++) {
            // 4. 현재 값으로 나눈 몫만큼 심볼 반복
            // why: num을 values[i]로 나눈 횟수만큼 해당 심볼 사용
            while (num >= values[i]) {
                result.append(symbols[i]);
                num -= values[i];
            }
        }
        
        // 5. 완성된 로마 숫자 문자열 반환
        // why: 모든 자릿수를 로마 숫자로 변환 완료
        return result.toString();
    }
}
```

**시간 복잡도**: O(1) - 값의 범위가 1~3999로 고정되어 있어 상수 시간  
**공간 복잡도**: O(1) - 결과 문자열 제외 시 고정된 크기의 배열만 사용

## 면접 대본 (영어)

"Let me explain my approach to converting an integer to a Roman numeral.

The key insight is to use a greedy approach with a predefined mapping of values to symbols, including the special subtractive cases. We process from the largest value to the smallest, using each symbol as many times as possible before moving to the next smaller value.

I'll create two parallel arrays: one for values and one for corresponding Roman symbols, ordered from largest to smallest. Critically, I include the special subtractive forms (900, 400, 90, 40, 9, 4) in this mapping alongside the standard symbols.

Here's how the algorithm works: I initialize a StringBuilder to efficiently build the result string. Then I iterate through each value-symbol pair. For each value, I check how many times it fits into the remaining number. I append the corresponding symbol that many times and subtract the value from num. I continue until num becomes zero.

The while loop for each value ensures we use that symbol as many times as possible before moving to the next smaller value. This greedy approach works because Roman numerals are designed to represent numbers by combining symbols from largest to smallest.

Let me walk through an example: num = 3749. Starting with value 1000 (M): 3749 >= 1000, append 'M', num = 2749. 2749 >= 1000, append 'M', num = 1749. 1749 >= 1000, append 'M', num = 749. 749 < 1000, move to next. Value 900 (CM): 749 < 900, skip. Value 500 (D): 749 >= 500, append 'D', num = 249. Value 400 (CD): 249 < 400, skip. Value 100 (C): 249 >= 100, append 'C', num = 149. 149 >= 100, append 'C', num = 49. Value 90 (XC): 49 < 90, skip. Value 50 (L): 49 < 50, skip. Value 40 (XL): 49 >= 40, append 'XL', num = 9. Value 10 (X): 9 < 10, skip. Value 9 (IX): 9 >= 9, append 'IX', num = 0. Result: "MMMDCCXLIX".

Another example: num = 58. Value 1000-100: all too large. Value 90: 58 < 90. Value 50 (L): 58 >= 50, append 'L', num = 8. Values 40, 10, 9: too large. Value 5 (V): 8 >= 5, append 'V', num = 3. Value 4: 3 < 4. Value 1 (I): 3 >= 1, append 'I', num = 2. 2 >= 1, append 'I', num = 1. 1 >= 1, append 'I', num = 0. Result: "LVIII".

A complex example: num = 1994. Value 1000 (M): append 'M', num = 994. Value 900 (CM): 994 >= 900, append 'CM', num = 94. Value 500-100: too large. Value 90 (XC): 94 >= 90, append 'XC', num = 4. Value 50-5: too large. Value 4 (IV): 4 >= 4, append 'IV', num = 0. Result: "MCMXCIV".

Why does including the subtractive forms in the mapping make this work? By treating combinations like 'CM' (900) as atomic units, we naturally handle the subtractive notation without special case logic. The greedy approach automatically selects these combinations when appropriate because they're positioned correctly in our ordered list.

An important detail: the order matters. We must process from largest to smallest to ensure we use larger denominations first. For example, 900 must come before 500 and 100, otherwise we'd incorrectly represent 900 as 'DCCCC' instead of 'CM'.

The time complexity is O(1) despite the loops because the input is bounded (1 to 3999), which means the maximum number of Roman numeral characters is fixed (at most 15 characters for 3888 = MMMDCCCLXXXVIII). The space complexity is also O(1) for the same reason - we use fixed-size arrays and the output size is bounded.

This problem elegantly demonstrates how a well-designed lookup table combined with a greedy approach can solve what initially seems like a complex conversion problem. The trick is recognizing that Roman numerals follow a greedy pattern and that including the special cases in our mapping eliminates the need for complex conditional logic."
