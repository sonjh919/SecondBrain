#Array #BinarySearch

## 풀이 코드
```java
class Solution {
    public int searchInsert(int[] nums, int target) {
        // 1. 이진 탐색을 위한 left, right 포인터 초기화
        // why: 정렬된 배열에서 O(log n) 탐색을 위해 이진 탐색 사용
        int left = 0;
        int right = nums.length - 1;
        
        // 2. left가 right를 넘어갈 때까지 탐색
        // why: 탐색 범위를 절반씩 줄여가며 target 위치 찾기
        while (left <= right) {
            // 3. 중간 인덱스 계산
            // why: 오버플로우 방지를 위해 left + (right - left) / 2 사용
            int mid = left + (right - left) / 2;
            
            // 4. target을 찾으면 해당 인덱스 반환
            // why: target이 배열에 존재하는 경우
            if (nums[mid] == target) {
                return mid;
            }
            // 5. target이 중간값보다 작으면 왼쪽 절반 탐색
            // why: 정렬된 배열에서 target은 왼쪽에 있음
            else if (nums[mid] > target) {
                right = mid - 1;
            }
            // 6. target이 중간값보다 크면 오른쪽 절반 탐색
            // why: 정렬된 배열에서 target은 오른쪽에 있음
            else {
                left = mid + 1;
            }
        }
        
        // 7. target을 찾지 못하면 left가 삽입 위치
        // why: left는 target보다 크거나 같은 첫 번째 위치를 가리킴
        return left;
    }
}
```

**시간 복잡도**: O(log n) - 이진 탐색으로 탐색 범위를 절반씩 줄임 (n은 배열 길이)  
**공간 복잡도**: O(1) - 포인터 변수만 사용

## 면접 대본 (영어)

"Let me explain my approach to finding the insert position in a sorted array.

The key insight is to use binary search since we need O(log n) time complexity and the array is sorted. Binary search works by repeatedly dividing the search space in half, comparing the target with the middle element, and eliminating half of the remaining elements based on that comparison.

The clever part about this problem is that when the target isn't found, the left pointer ends up pointing to exactly where the target should be inserted to maintain sorted order. This happens naturally as a result of how binary search works.

I'll use two pointers, left and right, to define the search range. Initially, left starts at 0 and right starts at the last index.

Here's how the algorithm works: In each iteration, I calculate the middle index using 'left + (right - left) / 2'. I use this formula instead of '(left + right) / 2' to avoid potential integer overflow, though with the given constraints it's not strictly necessary.

Then I compare the value at mid with the target. If they're equal, I've found the target, so I return mid immediately. If the middle value is greater than the target, the target must be in the left half, so I move right to mid - 1. If the middle value is less than the target, the target must be in the right half, so I move left to mid + 1.

The loop continues until left exceeds right. At this point, left points to the position where the target should be inserted.

Let me walk through an example: nums = [1,3,5,6], target = 5. Initially left=0, right=3. Mid=1, nums[1]=3 < 5, so left becomes 2. Now mid=2, nums[2]=5 equals target, so we return 2.

Another example: nums = [1,3,5,6], target = 2. Initially left=0, right=3. Mid=1, nums[1]=3 > 2, so right becomes 0. Now mid=0, nums[0]=1 < 2, so left becomes 1. Now left > right, we exit and return left=1. This is correct because 2 should be inserted at index 1, between 1 and 3.

A boundary case: nums = [1,3,5,6], target = 7. Left=0, right=3, mid=1, 3 < 7, left=2. Mid=3, 6 < 7, left=4. Now left > right, return left=4, which correctly represents inserting at the end.

Another boundary case: nums = [1,3,5,6], target = 0. Left=0, right=3, mid=1, 3 > 0, right=0. Mid=0, 1 > 0, right=-1. Now left > right, return left=0, correctly representing insertion at the beginning.

Why does left point to the insert position when target isn't found? During binary search, left always points to elements that are less than target, and right points to elements greater than or equal to target. When they cross, left is at the first position where target should be inserted to maintain sorted order.

The time complexity is O(log n) because we halve the search space in each iteration. Starting with n elements, we need at most log₂(n) iterations to narrow it down to a single position. The space complexity is O(1) since we only use a constant number of variables.

This solution elegantly handles both finding an existing element and determining the insertion position for a non-existent element in a single binary search pass."
