#LinkedList #Recursion

## 풀이 코드
```java
/**
 * Definition for singly-linked list.
 * public class ListNode {
 *     int val;
 *     ListNode next;
 *     ListNode() {}
 *     ListNode(int val) { this.val = val; }
 *     ListNode(int val, ListNode next) { this.val = val; this.next = next; }
 * }
 */
class Solution {
    public ListNode mergeTwoLists(ListNode list1, ListNode list2) {
        // 1. 더미 노드 생성하여 결과 리스트의 시작점 표시
        // why: 첫 노드 처리를 단순화하고 결과 리스트의 헤드를 쉽게 반환
        ListNode dummy = new ListNode(0);
        ListNode current = dummy;
        
        // 2. 두 리스트를 동시에 순회하며 작은 값을 결과에 추가
        // why: 정렬된 두 리스트를 병합하여 정렬된 하나의 리스트 생성
        while (list1 != null && list2 != null) {
            // 3. list1의 값이 작거나 같으면 list1 노드를 선택
            // why: 작은 값부터 결과 리스트에 추가하여 정렬 유지
            if (list1.val <= list2.val) {
                current.next = list1;
                list1 = list1.next;
            } 
            // 4. list2의 값이 작으면 list2 노드를 선택
            // why: 두 값을 비교하여 더 작은 값을 선택
            else {
                current.next = list2;
                list2 = list2.next;
            }
            current = current.next;
        }
        
        // 5. 남은 노드들을 결과 리스트에 연결
        // why: 한 리스트가 끝나면 다른 리스트의 나머지는 이미 정렬되어 있음
        if (list1 != null) {
            current.next = list1;
        }
        if (list2 != null) {
            current.next = list2;
        }
        
        // 6. 더미 노드 다음부터가 실제 병합된 리스트
        // why: 더미 노드는 편의상 사용한 것이므로 제외하고 반환
        return dummy.next;
    }
}
```

**시간 복잡도**: O(n + m) - 두 리스트의 모든 노드를 한 번씩 방문 (n, m은 각 리스트 길이)  
**공간 복잡도**: O(1) - 새 노드를 생성하지 않고 기존 노드를 재배치 (더미 노드만 생성)

## 면접 대본 (영어)

"Let me explain my approach to merging two sorted linked lists.

The key insight is that since both lists are already sorted, I can merge them in a single pass by always choosing the smaller value from the front of either list. This is similar to the merge step in merge sort.

I'll use a dummy node technique to simplify the code. The dummy node acts as a placeholder that points to the head of the result list, making it easier to handle edge cases and return the final result.

Here's how the algorithm works: I create a dummy node initialized with value 0, and a current pointer that starts at the dummy node. This current pointer will help me build the merged list.

Then I enter a loop that continues as long as both lists have remaining nodes. In each iteration, I compare the values at the front of list1 and list2. Whichever has the smaller value, I append that node to the result list by setting current.next to point to it. Then I advance the pointer of the list I took from, and also advance the current pointer to the newly added node.

This process continues until one of the lists is exhausted. At that point, I know that all remaining nodes in the other list are already sorted and greater than everything I've added so far, so I can simply attach the rest of that list to the end of my result.

Finally, I return dummy.next, which points to the actual head of the merged list, skipping over the dummy node.

Let me walk through an example: if we have list1 = [1,2,4] and list2 = [1,3,4], I start with dummy pointing to nothing. First comparison: 1 vs 1, they're equal, so I take from list1. Current.next points to the first 1 from list1. Second: 2 vs 1, list2's 1 is smaller, so I take it. Third: 2 vs 3, I take 2. Fourth: 4 vs 3, I take 3. Fifth: 4 vs 4, I take from list1. Finally, list1 is exhausted, so I attach the remaining 4 from list2. The result is [1,1,2,3,4,4].

Edge cases are handled elegantly: if list1 is empty, the while loop never executes, and I just attach list2 to dummy.next. Similarly for empty list2. If both are empty, dummy.next is null, which is correct.

The time complexity is O(n + m) where n and m are the lengths of the two lists, since we visit each node exactly once. The space complexity is O(1) because we're not creating new nodes - we're just rearranging pointers to existing nodes. The dummy node is a constant space overhead.

An alternative approach would be to use recursion, which is more elegant but uses O(n + m) space for the call stack. The iterative approach I've shown is more space-efficient and is generally preferred in production code to avoid potential stack overflow with very long lists."
