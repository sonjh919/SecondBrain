## 풀이 코드
```java
class Solution {
    public boolean containsNearbyDuplicate(int[] nums, int k) {
        // 1. 값과 최근 인덱스를 저장할 HashMap 생성
        // why: 각 숫자의 가장 최근 위치를 O(1) 시간에 조회하기 위함
        Map<Integer, Integer> map = new HashMap<>();
        
        // 2. 배열을 순회하며 중복과 거리 조건 확인
        // why: 현재 숫자가 이전에 나왔고 거리가 k 이하인지 검사
        for (int i = 0; i < nums.length; i++) {
            // 3. 현재 숫자가 이미 map에 있으면 거리 확인
            // why: 같은 값이 이전에 나왔다면 인덱스 차이를 계산
            if (map.containsKey(nums[i])) {
                // 4. 인덱스 차이가 k 이하면 조건 만족
                // why: 같은 값이 k 거리 안에 있으면 true 반환
                if (i - map.get(nums[i]) <= k) {
                    return true;
                }
            }
            
            // 5. 현재 숫자와 인덱스를 map에 저장 (업데이트)
            // why: 나중에 나올 같은 숫자와의 거리 계산을 위해 최신 위치 기록
            map.put(nums[i], i);
        }
        
        // 6. 조건을 만족하는 쌍을 찾지 못하면 false 반환
        // why: 모든 원소를 확인했지만 k 거리 안의 중복이 없음
        return false;
    }
}
```

**시간 복잡도**: O(n) - 배열을 한 번 순회 (n은 배열 길이)  
**공간 복잡도**: O(min(n, k)) - 최대 k+1개의 고유한 원소를 저장 (sliding window 효과)

## 면접 대본 (영어)

"Let me explain my approach to finding nearby duplicates within a certain distance.

The key insight is to use a HashMap to store each number along with its most recent index. As we traverse the array, whenever we encounter a number we've seen before, we check if the distance between the current index and the previously stored index is within k.

I'll use a HashMap where the key is the number and the value is its most recent index. This allows me to check both existence and distance in constant time.

Here's how the algorithm works: I iterate through the array with index i. For each element, I first check if this number already exists in the HashMap. If it does, I calculate the distance between the current index i and the stored index. If this distance is less than or equal to k, I've found a nearby duplicate, so I return true immediately.

Regardless of whether we found a match, I update the HashMap with the current number and its index. This is important because we always want to store the most recent occurrence - this maximizes our chances of finding a valid pair in future iterations, since we're looking for the closest duplicates.

Let me walk through an example: if we have nums = [1,2,3,1] and k = 3, at i=0, we see 1 and add 1->0 to the map. At i=1, we see 2 and add 2->1. At i=2, we see 3 and add 3->2. At i=3, we see 1 again. We check the map and find 1 at index 0. The distance is 3-0=3, which is exactly k, so we return true.

Another example: if we have nums = [1,0,1,1] and k = 1, at i=0, we add 1->0. At i=1, we add 0->1. At i=2, we see 1 again. The distance is 2-0=2, which is greater than k=1, so we don't return yet. We update the map to 1->2. At i=3, we see 1 again. The distance is 3-2=1, which equals k, so we return true.

A failing case: nums = [1,2,3,1,2,3] and k = 2. At i=0,1,2, we add entries. At i=3, we see 1 at distance 3-0=3 > k=2, so we update to 1->3. At i=4, we see 2 at distance 4-1=3 > k=2, update to 2->4. At i=5, we see 3 at distance 5-2=3 > k=2, update to 3->5. We never find a pair within distance k, so we return false.

The time complexity is O(n) where n is the length of the array, since we make a single pass and each HashMap operation is O(1) on average. The space complexity is interesting - it's O(min(n, k)). While we might think we need O(n) space for all unique elements, in practice, if we've gone more than k steps without seeing a duplicate, those old entries become irrelevant. However, my current implementation doesn't remove old entries, so it's technically O(n) worst case. An optimization would be to use a sliding window with a HashSet and remove elements that are more than k positions away, achieving true O(k) space."

## 핵심 아이디어

HashMap에 각 숫자의 최근 인덱스를 저장하고 중복 발견 시 거리를 확인하여 O(n) 시간에 k 거리 내 중복 판별
