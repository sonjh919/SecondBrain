#HashTable #Math #TwoPointers

## 풀이 코드
```java
class Solution {
    public boolean isHappy(int n) {
        // 1. 이미 본 숫자를 저장할 HashSet 생성
        // why: 사이클을 감지하기 위해 이전에 나온 숫자들을 기록
        Set<Integer> seen = new HashSet<>();
        
        // 2. n이 1이 되거나 사이클이 발견될 때까지 반복
        // why: happy number는 결국 1이 되고, 아니면 사이클에 빠짐
        while (n != 1 && !seen.contains(n)) {
            seen.add(n);
            n = getNext(n);
        }
        
        // 3. 1이면 happy number, 아니면 사이클에 빠진 것
        // why: 루프를 벗어났을 때 n==1이면 성공, 아니면 사이클 발견
        return n == 1;
    }
    
    // 4. 각 자릿수의 제곱의 합을 계산하는 헬퍼 함수
    // why: 다음 숫자를 계산하는 로직을 분리하여 코드 가독성 향상
    private int getNext(int n) {
        int sum = 0;
        while (n > 0) {
            int digit = n % 10;
            sum += digit * digit;
            n /= 10;
        }
        return sum;
    }
}
```

**시간 복잡도**: O(log n) - 각 단계에서 자릿수만큼 연산, 숫자는 빠르게 작아짐  
**공간 복잡도**: O(log n) - HashSet에 저장되는 숫자의 개수 (사이클 크기는 제한적)

## 면접 대본 (영어)

"Let me explain my approach to determining if a number is happy.

The key insight is that this process will either reach 1, making it a happy number, or it will enter a cycle and never reach 1. We need to detect this cycle to know when to stop.

I'll use a HashSet to track all the numbers we've seen during the process. If we encounter the same number twice, it means we've entered a cycle and will never reach 1.

The algorithm works like this: I start with the given number n. I repeatedly calculate the sum of the squares of its digits using a helper function getNext. Before each calculation, I check two conditions: first, if n equals 1, we've found a happy number. Second, if n is already in our seen set, we've detected a cycle. If neither condition is met, I add n to the seen set and continue with the next number.

The helper function getNext takes a number and calculates the sum of squares of its digits. I extract each digit using modulo 10, square it, add it to the sum, then divide the number by 10 to move to the next digit. This continues until all digits are processed.

Let me walk through an example: if we have n = 19, first iteration gives us 1² + 9² = 1 + 81 = 82. We add 19 to seen. Second iteration: 8² + 2² = 64 + 4 = 68. We add 82 to seen. Third iteration: 6² + 8² = 36 + 64 = 100. We add 68 to seen. Fourth iteration: 1² + 0² + 0² = 1. We've reached 1, so we return true.

Another example: n = 2. First: 2² = 4. Second: 4² = 16. Third: 1² + 6² = 37. Fourth: 3² + 7² = 58. Fifth: 5² + 8² = 89. Sixth: 8² + 9² = 145. Continuing this process, we eventually see 4 again, which means we've entered a cycle that doesn't include 1, so we return false.

The time complexity is tricky to analyze. For each number, calculating the next number takes O(log n) time where log n is the number of digits. However, the numbers quickly get smaller - for example, the largest three-digit number 999 becomes 243. Research shows that all numbers either reach 1 or enter a cycle with very small numbers, so the process terminates quickly. The practical time complexity is O(log n).

The space complexity is O(log n) for the HashSet. While we might worry about storing many numbers, the cycle sizes are actually quite small in practice - typically less than 10 numbers.

An alternative approach would use Floyd's cycle detection algorithm with two pointers - a slow and fast pointer - which would reduce space complexity to O(1), but the HashSet approach is more intuitive and efficient enough for this problem."
