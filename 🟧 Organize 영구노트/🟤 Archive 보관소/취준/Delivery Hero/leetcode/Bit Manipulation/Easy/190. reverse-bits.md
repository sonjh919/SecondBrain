#BitManipulation #DivideAndConquer

## 풀이 코드
```java
class Solution {
    public int reverseBits(int n) {
        // 1. 결과를 저장할 변수 초기화
        // why: 비트를 역순으로 쌓아서 최종 결과를 만듦
        int result = 0;
        
        // 2. 32번 반복하여 모든 비트 처리
        // why: 32비트 정수의 모든 비트를 역순으로 배치해야 함
        for (int i = 0; i < 32; i++) {
            // 3. result를 왼쪽으로 1비트 이동하여 자리 확보
            // why: 새로운 비트를 추가할 공간을 만듦
            result <<= 1;
            
            // 4. n의 최하위 비트를 result에 추가
            // why: n & 1로 최하위 비트를 추출하고 OR 연산으로 result에 추가
            result |= (n & 1);
            
            // 5. n을 오른쪽으로 1비트 이동
            // why: 다음 비트를 처리하기 위해 현재 최하위 비트 제거
            n >>>= 1;
        }
        
        // 6. 역순으로 배치된 결과 반환
        // why: 모든 비트가 역순으로 재배치됨
        return result;
    }
}
```

**시간 복잡도**: O(1) - 항상 32번의 고정된 반복 수행  
**공간 복잡도**: O(1) - 고정된 개수의 변수만 사용

## 면접 대본 (영어)

"Let me explain my approach to reversing the bits of a 32-bit integer.

The key insight is to process the bits one at a time from right to left in the input, and build the result from left to right. We extract the least significant bit from n, add it to result, shift result left to make room for the next bit, and shift n right to get the next bit. We repeat this 32 times to process all bits.

I'll use bit manipulation operations: left shift to move bits left, right shift to move bits right, AND to extract specific bits, and OR to add bits to the result.

Here's how the algorithm works: I initialize result to 0. Then I loop 32 times, once for each bit in a 32-bit integer. In each iteration, I first shift result left by 1 bit using the left shift operator. This makes room for the new bit we're about to add.

Next, I extract the least significant bit of n using 'n & 1'. The AND operation with 1 gives us the rightmost bit. Then I add this bit to result using the OR operator. The OR operation sets the least significant bit of result to whatever we extracted from n.

Finally, I shift n right by 1 bit using the unsigned right shift operator '>>>'. I use unsigned shift instead of regular shift to ensure we fill with zeros on the left, not with the sign bit. This moves the next bit into the least significant position for the next iteration.

After 32 iterations, result contains all bits from n in reverse order.

Let me walk through a small example with just 8 bits for clarity. Suppose n = 43 in binary: 00101011. Initially result = 00000000. Iteration 1: result shifts left to 00000000, n's last bit is 1, OR gives result = 00000001, n becomes 00010101. Iteration 2: result = 00000010, n's last bit is 1, result = 00000011, n = 00001010. Continuing this way for 8 iterations, we'd get 11010100, which is the reverse of 00101011.

An important detail: I use unsigned right shift (>>>) instead of signed right shift (>>) for n. This is crucial because if n has the sign bit set (appears negative), a signed shift would propagate the sign bit, but we want to treat it as an unsigned 32-bit number and fill with zeros.

For the actual example: n = 43261596 in binary is 00000010100101000001111010011100. After reversing all 32 bits, we get 00111001011110000010100101000000, which equals 964176192.

The time complexity is O(1) because we always do exactly 32 iterations regardless of the input value. The space complexity is O(1) because we only use a constant amount of extra space for the result variable.

For the follow-up about optimization if called many times: we could use memoization with a cache. Since we're dealing with 32-bit integers, we could cache results for frequently occurring values. Another approach is divide-and-conquer: reverse pairs of bits, then reverse 2-bit groups, then 4-bit groups, and so on. This is called bit reversal by successive doubling and can be faster for batch processing, though it's more complex to implement. We could also precompute reversed values for all possible 8-bit numbers (256 values), then split the 32-bit number into four 8-bit chunks and combine their reversals."
