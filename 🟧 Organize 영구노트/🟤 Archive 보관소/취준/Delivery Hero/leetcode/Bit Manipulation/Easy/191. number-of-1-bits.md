#BitManipulation

## 풀이 코드
```java
class Solution {
    public int hammingWeight(int n) {
        // 1. 1 비트의 개수를 저장할 변수 초기화
        // why: set bit의 개수를 카운트하기 위함
        int count = 0;
        
        // 2. n이 0이 될 때까지 반복
        // why: 모든 1 비트를 제거할 때까지 계속
        while (n != 0) {
            // 3. Brian Kernighan's Algorithm - 최하위 1 비트 제거
            // why: n & (n-1)은 가장 오른쪽의 1 비트를 0으로 만듦
            n = n & (n - 1);
            count++;
        }
        
        // 4. 1 비트의 총 개수 반환
        // why: 반복 횟수가 곧 set bit의 개수
        return count;
    }
}
```

**시간 복잡도**: O(k) - k는 1 비트의 개수 (최대 32번)  
**공간 복잡도**: O(1) - 카운트 변수만 사용

## 면접 대본 (영어)

"Let me explain my approach to counting the number of 1 bits in an integer.

The key insight is to use Brian Kernighan's Algorithm, which is an elegant bit manipulation trick. Instead of checking every bit position, we can directly count only the 1 bits by repeatedly removing the rightmost 1 bit until the number becomes zero.

The magic happens with the operation 'n & (n-1)'. Let me explain why this removes the rightmost 1 bit. When we subtract 1 from a number, all bits after the rightmost 1 (including that 1) get flipped. For example, if n is 1011000, then n-1 is 1010111. When we AND these together, the rightmost 1 and all zeros to its right become 0, giving us 1010000.

Here's how the algorithm works: I initialize a count to zero. Then I loop while n is not zero. In each iteration, I perform 'n & (n-1)' which removes the rightmost 1 bit from n, and increment the count. The number of iterations equals the number of 1 bits, because each iteration removes exactly one 1 bit.

Let me walk through an example: n = 11, which is 1011 in binary. First iteration: n & (n-1) = 1011 & 1010 = 1010, count becomes 1. Second iteration: 1010 & 1001 = 1000, count becomes 2. Third iteration: 1000 & 0111 = 0000, count becomes 3. Now n is 0, we exit and return 3.

Another example: n = 128, which is 10000000 in binary. First iteration: 10000000 & 01111111 = 00000000, count becomes 1. Now n is 0, return 1. Notice how efficiently this works - we only did one iteration because there's only one 1 bit.

A more complex example: n = 2147483645, which is 1111111111111111111111111111101 - thirty 1s and one 0. The algorithm will iterate exactly 30 times, once for each 1 bit, regardless of where the 0 is positioned.

Why is this algorithm superior to the naive approach? A naive solution would check all 32 bits using a loop with right shifts and checking 'n & 1'. That always takes 32 iterations. Brian Kernighan's algorithm only iterates once per 1 bit, so it's optimal - we do the minimum possible work.

The time complexity is O(k) where k is the number of 1 bits in n. In the best case where n has only one 1 bit, it's O(1). In the worst case where all 32 bits are 1, it's O(32) = O(1) since 32 is a constant. The space complexity is O(1) because we only use a counter variable.

For the follow-up about optimization if called many times: we could use lookup tables. One approach is to precompute the number of 1 bits for all possible 8-bit values (256 entries), then split the 32-bit number into four 8-bit chunks and sum up the precomputed results. This trades space for time, using O(256) = O(1) space to achieve O(4) = O(1) time per call. Another approach is memoization if we expect repeated values, though with 32-bit integers the cache would be huge. The lookup table approach is generally most practical.

An interesting note: many modern CPUs have a built-in instruction for this operation called POPCNT (population count), which counts set bits in hardware. In Java, you could use Integer.bitCount(n) which likely uses this instruction when available, but understanding the algorithmic approach is valuable for interviews and for languages without such built-ins."
