# 리뷰 체크리스트

## 우선순위
1. 책임 분리와 객체 설계 — 설계가 잘못되면 네이밍/테스트도 다 바뀜. 영향도가 가장 큼
2. 네이밍 — 행위와 의도가 이름에 드러나는지, 사용 맥락이 이름에 새지 않는지
3. 테스트 커버리지와 품질 — 실패/엣지 케이스 포함, 환경 독립성
4. 코드 품질 기본 — 매직 넘버, final, 포매팅, 접근 제한자
5. 성능 — 명백한 낭비만 짚음 (반복 객체 생성, N+1 쿼리)

## 필수 확인 TOP 5
1. 책임 분리 — 생성자는 생성만, 도메인 결정은 도메인에서, View가 도메인 로직을 품지 않는가
2. 상속 적절성 — 재사용→조합, 타입계층→추상클래스, 일반 메서드 오버라이딩 지양, 분류 기준 일관성
3. 네이밍 — 행위/의도를 드러내는가, 사용 맥락이 이름에 새지 않는가
4. 테스트 — 주요 도메인 로직 단위 테스트 + 실패/엣지 케이스 포함
5. 원시 타입 포장 — 도메인 의미가 있는 값은 전용 타입으로

## 추가 확인
### 코드 품질
- [ ] 매직 넘버가 상수로 관리되는가
- [ ] final 키워드가 적절히 사용되었는가
- [ ] 포매팅/코드 스타일이 일관적인가
- [ ] 접근 제한자가 최소 범위로 설정되었는가
- [ ] 불필요한 import나 사용하지 않는 코드가 없는가

### 설계/아키텍처
- [ ] 패키지 구조가 도메인을 반영하는가
- [ ] 함수형 인터페이스 사용 시 커스텀 타입으로 의도를 드러내는가

### 안전성
- [ ] 예외가 적절한 위치에서 처리되는가
- [ ] 외부 의존성에 대한 테스트가 격리되었는가

### 테스트
- [ ] 테스트가 환경에 독립적인가
- [ ] 테스트 간 독립성이 보장되는가

## 톤
- 질문형 피드백으로 생각을 유도한다 ("왜 이렇게 하셨나요?")
- 근거 자료(링크, 코드 예시)를 함께 제공한다
- 잘한 부분을 먼저 인정하고 개선점을 제안한다
- "정답"보다 "왜?"를 묻고 직접 구현해보도록 제안한다
