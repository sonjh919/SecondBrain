## 리뷰 원칙

내가 코드를 리뷰할 때 지키는 핵심 원칙들. 직접 겪은 경험에서 내재화한 것들이다.

### 1. 네이밍이 의도를 정확히 전달하는지 확인한다
메서드명, 변수명, 파라미터명이 **행위나 역할**을 드러내는지 본다. `getXXX`보다 행위가 드러나는 이름, `from()`의 무의식적 남용, 범용적 이름(`Amount`, `Input`) 대신 맥락을 담은 이름을 기대한다.

> **경험 근거**: "가급적 getXXX, setXXX 보다는 메서드의 행위가 드러나는 네이밍을 고민해보시면 좋을 것 같아요." — PR #521. 이후 "프리코스 때 from으로 네이밍을 했었던 경험이 무의식적으로 다시 튀어나온것 같네요"라며 관성적 네이밍의 위험을 직접 체감함.
> **리뷰 시 적용**: "이 메서드/변수 이름만 보고 역할을 알 수 있나요?"라고 질문형으로 짚어준다. 대안 네이밍을 함께 제시한다.

### 2. 생성자/메서드가 하나의 역할만 하는지 확인한다
생성자에서 파싱, 검증, 변환, 생성을 모두 수행하면 역할이 과부하된다. 특히 문자열 파싱 같은 외부 입력 처리가 도메인 객체 생성자에 섞이면 변경 전파의 원인이 된다.

> **경험 근거**: "생성자는 가급적 객체를 생성하는 역할만 수행하도록해보면 어떨까요? 문자열 입력을 받아 ,로 분리하고, 각 문자를 숫자로 변환하는 이런 일련의 과정은 외부에서 수행하여 생성자 파라미터로 전달해주도록해보면 좋을 것 같습니다." — PR #521. 이를 정적 팩토리 메서드로 분리하여 해결하는 경험을 직접 함.
> **리뷰 시 적용**: 생성자가 3가지 이상의 일을 하고 있다면 "이 중 생성과 무관한 로직은 외부로 빼볼 수 있을까요?"라고 제안한다.

### 3. 실수를 코드로 방지할 수 있는지 확인한다
`final`, 접근 제한자, 방어적 복사 등 언어가 제공하는 안전장치를 활용하고 있는지 본다. "나는 실수 안 해"가 아니라 "실수해도 컴파일러가 잡아주는 구조"를 지향한다.

> **경험 근거**: "이건 붙이라고 했어!라는 마인드로 쓰는건 바람직하지 않다"는 피드백에, "사람은 항상 실수할 수 있기 때문에, 최대한 언어쪽에서 보장해주는 안전장치가 필요하다는 말씀이 맞는 것 같아요"라며 납득. — PR #521
> **리뷰 시 적용**: final 누락, public 상수 노출, 방어적 복사 미적용 등을 발견하면 **왜 필요한지 이유와 함께** 짚어준다. 단순히 "final 붙이세요"가 아니라 실수 방지의 맥락을 전달한다.

### 4. 도메인 로직이 적절한 위치에 있는지 확인한다
도메인에 종속된 코드가 `global`, `util`, `common` 같은 범용 패키지에 빠져 있지 않은지 본다. 패키지 구조가 도메인의 응집도를 반영하는지 확인한다.

> **경험 근거**: "global이라는 패키지는 어떤 경우를 의미하는걸까요? 예외 메시지도 생각보다 로또 도메인에 종속된 내용들이 많아서 조금 혼란이 생기는 것 같아요." — PR #521. 이후 "도메인에 종속된다는 의미를 제가 너무 간과해서 생각했던 점이 있는 것 같습니다"라며 교훈 얻음.
> **리뷰 시 적용**: "이 클래스/상수가 실제로 여러 도메인에서 쓰이나요? 아니면 특정 도메인에만 관련된 건가요?"라고 질문한다.

### 5. 테스트가 핵심 로직을 커버하는지 확인한다
"내부적으로 호출되니까 괜찮다"는 논리로 테스트를 생략하는 것을 경계한다. 핵심 비즈니스 로직은 독립적으로 테스트되어야 리팩토링 시 안전망이 된다.

> **경험 근거**: "내부적으로 호출하기 때문에 어느정도 동작함을 보장할 수도 있지만, 완벽하게 커버되는 것은 아니라고 생각해요. 이 테스트 코드가 통과하는지만 확인하면 이전과 동일하게 동작하는지를 보장받을 수도 있기 때문에 리팩터링을 맘편히할 수도 있다고 생각해요." — PR #661
> **리뷰 시 적용**: 핵심 로직에 대한 테스트가 빠져 있으면 "이 메서드에 대한 독립 테스트가 있으면 이후 리팩토링할 때 안전망이 될 것 같아요"라고 제안한다.

### 6. 트레이드오프를 의식하고 있는지 확인한다
설계 결정에서 "왜 이쪽을 선택했는지"가 명확한지 본다. 양쪽의 장단점을 비교하지 않고 한쪽만 따르는 코드는 질문한다.

> **경험 근거**: "요구사항을 한 곳에 몰아넣어 유지보수를 편리하게 하는 설계 vs 컨트롤 불가능한 동작을 외부에서 주입받는 설계 두 가지 경우에서 트레이드오프를 고려해야 할 것 같은데" — PR #661. 스스로 트레이드오프를 고민하고, 리뷰어에게 검증받는 방식을 체득함.
> **리뷰 시 적용**: 설계 결정이 보이면 "이 방식을 선택한 이유가 있을까요?"라고 물어 사고 과정을 확인한다. 답이 명확하면 존중하고, 불명확하면 대안을 제시한다.

## 리뷰 체크리스트

코드를 리뷰할 때 카테고리별로 확인할 항목들.

### 코드 품질
- [ ] **네이밍이 행위/역할을 드러내는가** — `getXXX` 대신 의미 있는 동사, 범용적 이름 대신 맥락을 담은 이름
  > "LottoFactory.from()의 결과로 Lotto 객체가 생성되는군요? 이 메서드는 결국 로또를 생성하는 행위를 하는 것이니 generate, create와 같은 네이밍이 더 맞지 않을까요?" — PR #521
- [ ] **매직 넘버가 상수로 추출되었는가** — 숫자 리터럴이 코드에 직접 들어가 있으면 의미를 알 수 없음
  > "1, 45, 6은 각각 어떤 숫자일까요? 👀" — PR #521
- [ ] **불필요한 변수 선언이 없는가** — 바로 반환하거나 인라인할 수 있는 변수
- [ ] **메서드 체이닝이 가독성을 해치지 않는가** — 한 줄에 과도하게 나열하지 않기

### 설계/아키텍처
- [ ] **생성자가 생성 역할만 하는가** — 파싱, 변환 로직이 생성자에 섞여 있지 않은지
- [ ] **도메인 코드가 도메인 패키지에 있는가** — global/util/common에 도메인 종속 코드가 빠져 있지 않은지
- [ ] **불필요한 클래스 분리가 없는가** — 하나로 합칠 수 있는 Factory/Handler가 무의식적으로 분리되어 있지 않은지
- [ ] **상속보다 합성을 고려했는가** — 상속이 결합도를 높이고 캡슐화를 깨뜨리는 경우 합성 제안
  > "현재 상속 대신 WinningLotto의 필드에 Lotto를 포함시키는 합성을 이용하여 리팩토링을 진행하고 있습니다." — PR #521 (직접 깨달은 교훈)
- [ ] **원시 타입 포장을 고려했는가** — 비즈니스 규칙이 있는 원시 타입(로또 번호 범위 등)은 별도 타입으로
  > "로또 숫자를 상징하는 타입을 정의해볼 수는 없을까요? 원시 타입을 포장해보면 어떨까요?" — PR #661

### 안전성
- [ ] **필드에 final이 적용되었는가** — 재할당되지 않는 필드는 final로 보호
- [ ] **접근 제한자가 최소 범위인가** — 내부에서만 쓰이는 상수가 public으로 노출되어 있지 않은지
- [ ] **방어적 복사가 적용되었는가** — 컬렉션을 외부에 반환할 때 `new ArrayList<>()`로 복사
- [ ] **검증이 비즈니스 로직으로서 적절한 위치에 있는가** — 정적 팩토리 메서드에서 검증 후 private 생성자로 위임

### 테스트
- [ ] **핵심 비즈니스 로직에 독립 테스트가 있는가** — "내부적으로 호출되니까"로 생략하지 않기
- [ ] **테스트 불가능한 동작이 인터페이스로 분리되었는가** — 랜덤, 시간 등 제어 불가능한 의존성
- [ ] **함수형 인터페이스 + 람다로 테스트 코드를 간결하게 할 수 있는가** — FakeXXX 클래스 대신 람다
  > "FakeGenerator를 사용한 테스트 코드에 람다식을 활용해서 테스트가 동작하도록 바꿔보시면 좋을 것 같습니다." — PR #521
- [ ] **테스트 메서드명이 무엇을 테스트하는지 명확한가** — 한글 메서드명 또는 @DisplayName 활용

### 기타
- [ ] **DTO 변환 로직이 DTO 안에 있는가** — 도메인 객체가 DTO를 알지 않도록
- [ ] **README에 기능 목록이 체크리스트로 정리되어 있는가** — 커밋 단위와 연결

## 리뷰 우선순위

코드를 볼 때 내가 먼저 보는 순서:

1. **설계 구조** — 가장 예민하게 보는 것. 역할 분리, 도메인 응집, 상속 vs 합성 등 구조적 결정. 직접 상속→합성 리팩토링을 겪으며 구조의 중요성을 체감했기 때문.
2. **네이밍** — 두 번째로 중요하게 보는 것. 6회 이상 지적받으며 "네이밍은 무의식적 관성이 가장 잘 드러나는 곳"임을 체득. 네이밍만 봐도 코드 작성자의 의도 파악 수준을 알 수 있음.
3. **안전장치 (final, 접근 제한자, 방어적 복사)** — 세 번째. "사람은 실수하니 언어가 잡아주는 구조"를 지향. 사소해 보이지만 협업 환경에서 중요.
4. **테스트 커버리지** — 네 번째. 핵심 로직의 독립 테스트 존재 여부. 리팩토링 안전망.
5. **코드 스타일 (매직 넘버, 불필요한 변수 등)** — 마지막. 중요하지만 위의 것들보다 우선순위는 낮음.

## 리뷰 톤 가이드

내가 지향하는 리뷰 커뮤니케이션 방식. Rok93의 리뷰 스타일에서 배운 것들이다.

### 본받을 방식
- **질문형 피드백**: 단정하지 않고 "~하면 어떨까요?"로 제안하여 상대의 사고를 유도한다.
  > "가급적 getXXX, setXXX 보다는 메서드의 행위가 드러나는 네이밍을 고민해보시면 좋을 것 같아요. 😃" — PR #521
- **근거 자료 첨부**: 주장만 하지 않고 관련 글, 레퍼런스를 함께 제시한다.
  > "의미가 불분명한 매직 넘버를 상수로 선언하라. (링크)", "원시 타입을 포장해보면 어떨까요? (링크)" — PR #521, #661
- **코드 제안 포함**: 말로만 하지 않고 `suggestion` 블록으로 구체적 대안을 보여준다.
  > suggestion 블록으로 직접 수정 코드를 제시하는 패턴 반복 — PR #521
- **칭찬과 지적의 균형**: 잘한 부분을 먼저 인정하고 개선점을 제안한다.
  > "링크 피드백 반영 잘해주셨습니다. 👍💯 이번 미션 요구사항에는... 테스트가 누락된 로직들을 확인해서 테스트 추가 부탁드릴게요. 🙏" — PR #521
- **맥락 설명**: "왜 이렇게 해야 하는지"를 설명하여 상대가 원칙을 이해하고 내재화할 수 있게 한다.
  > "추후 입력 방식이 바뀐다거나하는 변경이 발생했을 때, 도메인로직까지 이 변경이 전파되는 문제도 자연스럽게 생길 것 같구요." — PR #521

### 피드백 시 주의할 점
- "~하세요"보다 "~해보면 어떨까요?"를 사용한다
- 지적만 하지 말고 왜 그래야 하는지 맥락을 반드시 포함한다
- 잘한 점을 구체적으로 짚어준다 (막연한 "잘했어요" 지양)
- 상대의 의도를 먼저 물어본다 — "이렇게 하신 이유가 있으신가요?"
- 참고 자료가 있다면 링크로 함께 제시한다
