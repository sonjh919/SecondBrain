## 리뷰 원칙

내가 코드를 리뷰할 때 지키는 핵심 원칙들. 직접 겪은 경험에서 내재화한 것들이다.

### 1. 네이밍은 행위와 의도를 드러내야 한다

코드에서 가장 먼저 눈에 들어오는 것은 이름이다. 클래스명, 메서드명, 변수명이 그 역할을 명확히 드러내지 못하면 코드를 읽는 모든 사람의 시간을 잡아먹는다. `getXXX`보다는 행위가 드러나는 이름, `from`보다는 `generate/create`처럼 동작의 의미가 정확한 이름, 범용적인 `Amount`보다는 구체적인 이름을 사용해야 한다. 또한 객체가 자신의 사용 맥락까지 알고 있는 이름(예: `firstOpenCards`)은 캡슐화를 해치므로, 순수한 책임만 드러나는 이름(예: `getFirstCard`)으로 바꿔야 한다.

> **경험 근거**: "getXXX, setXXX 보다는 메서드의 행위가 드러나는 네이밍을 고민해보시면 좋을 것 같아요." — PR #521 (로키)
> "Amount는 로또 구입 갯수를 의미하는걸까요? 네이밍에서 그 의미가 드러나면 좋을 것 같아요." — PR #521 (로키)
> "Hand가 자신이 어떻게 쓰일지 맥락을 알고 있는 것 같은 느낌이 들어서 남긴 거였어요. '공개'라는 맥락은 딜러와 플레이어만 알면 될 것 같고" — PR #853 (범블비)
> **리뷰 시 적용**: 이름만 보고 역할을 짐작할 수 없다면, 더 나은 이름을 제안한다. 특히 메서드명이 동사가 아니거나 객체의 사용 맥락이 이름에 녹아있는 경우 짚어준다.

### 2. 객체의 책임을 명확히 분리하라

하나의 객체가 여러 역할을 수행하면 코드를 이해하기 어렵고, 변경 시 영향 범위가 넓어진다. 생성자는 객체 생성의 역할만, 도메인 로직은 해당 도메인 객체 안에서, View 관련 결정은 View가 아닌 도메인에서 판단해야 한다. Deck과 Hand처럼 같은 자료구조를 쓰더라도 책임이 다르면 분리해야 한다.

> **경험 근거**: "Hand 객체는 따로 만들어도 좋을 것 같아요. Deck에서의 책임과 Hand에서의 책임이 완전 다른 것 같네요." — PR #808 (범블비)
> "생성자는 가급적 객체를 생성하는 역할만 수행하도록해보면 어떨까요?" — PR #521 (로키)
> "딜러는 처음에 한 장의 카드만 공개한다는 도메인 요구사항인 것 같아요. 지금은 View에서 결정하고 있네요." — PR #808 (범블비)
> **리뷰 시 적용**: 하나의 클래스에 서로 다른 성격의 메서드가 섞여 있으면, 책임 분리를 제안한다. 특히 View가 도메인 결정을 하고 있거나, 생성자가 파싱/검증까지 수행하는 경우 지적한다.

### 3. 상속보다 조합을 우선 고려하되, 타입 계층이 필요하면 추상 클래스를 쓰라

일반 클래스의 상속은 결합도를 높이고 예측하기 어렵다. 재사용 목적이라면 조합이 훨씬 안전하다. 하지만 요구사항을 타입으로 표현해야 할 때(예: "첫 공개 카드를 알려줘"라는 메시지를 딜러와 플레이어가 다르게 응답)는 추상 클래스와 추상 메서드가 적절하다. 일반 메서드의 오버라이딩은 예측하기 어려우므로 지양한다. 중간 타입 계층의 분류 기준은 일관되어야 한다.

> **경험 근거**: "일반 클래스의 상속은 지양해주세요. 단순하게 재사용의 목적만 있다면 조합이 훨씬 안전한 방법이죠." — PR #808 (범블비)
> "추상 메서드가 아닌 일반 메서드의 오버라이딩은 예측하기 어려워서 지양해야합니다." — PR #853 (범블비)
> "일부는 네이밍에서부터 기존과 다르게 Type에 따른 역할 구분이 되어있다고 느껴졌어요. Piece가 최상위인데, 그 하위 객체들은 일관된 기준이 아닌 서로 다른 기준으로 나뉜 것 같아요." — PR #102 (조앤)
> **리뷰 시 적용**: 상속을 사용한 코드를 보면, (1) 재사용 목적인지 타입 계층 목적인지 확인하고, (2) 일반 메서드 오버라이딩이 있는지 확인하고, (3) 분류 기준이 일관되는지 확인한다.

### 4. 테스트는 동작을 검증하고, 엣지 케이스를 포함해야 한다

테스트는 구현이 아니라 동작을 검증하는 것이다. 내부에서 호출되는 메서드라도 독립적인 테스트가 있으면 리팩터링 시 안전망이 된다. 성공 케이스뿐 아니라 실패/예외/엣지 케이스도 반드시 포함해야 한다. 테스트 환경은 누구의 로컬에서든 동작해야 한다.

> **경험 근거**: "내부적으로 호출하기 때문에 어느정도 이 메서드가 동작함을 보장할 수도 있지만... 추후 팀원이 이 메서드를 수정하더라도 이 테스트 코드가 통과하는지만 확인하면... 리팩터링을 맘편히할 수도 있다" — PR #661 (로키)
> "잘 가는지도 중요하지만, 잘 안가는지도 중요하닷" — PR #80 (조앤)
> "테스트를 전체적으로 실행하면 실패하는 테스트들이 존재해요... 어떤 사람의 환경이라도 테스트가 성공하도록 하려면 어떻게 할 수 있을까요?" — PR #102 (조앤)
> **리뷰 시 적용**: 주요 메서드에 테스트가 있는지, 실패 케이스가 포함되었는지, 특정 환경에 의존하지 않는지 확인한다.

### 5. 도메인을 이해하고, 근거 있는 결정을 하라

도메인을 모르고 구현하면 엣지 케이스를 놓치고, 설계에는 항상 트레이드오프가 따른다. 명확하지 않은 요구사항은 직접 조사해서 명확하게 만들어야 하고, 모든 설계 결정에는 "왜 이렇게 했는지"에 대한 근거가 있어야 한다. 상상만으로 판단하지 말고 직접 구현해보고 비교해야 한다.

> **경험 근거**: "도메인을 알고 구현하는 것과 모르고 구현하는 건 천지차이입니다." — PR #808 (범블비)
> "명확하지 않은 요구사항을 명확하게 만들어가는 것도 개발자의 역량입니다." — PR #808 (범블비)
> "모든 방식에는 트레이드오프가 발생합니다" — PR #808 (범블비)
> "별로라고 생각하지만 막상 구현해보면 코드가 훨씬 잘 읽히는 케이스도 많습니다." — PR #808 (범블비)
> **리뷰 시 적용**: 도메인 로직이 현실과 맞지 않거나 요구사항을 대충 넘긴 흔적이 보이면 짚어준다. 설계 결정에 대해 "왜?"를 물어보고, 근거가 모호하면 직접 구현해보도록 제안한다.

## 리뷰 우선순위

코드를 볼 때 내가 먼저 보는 순서:

1. **책임 분리와 객체 설계** — 가장 영향도가 큰 영역. 생성자 역할, 도메인 vs View 결정권, 상속 vs 조합 등. 설계가 잘못되면 수정 비용이 기하급수적으로 커지고, 네이밍이나 테스트는 설계가 바뀌면 함께 바뀌기 때문에 설계를 먼저 본다.
2. **네이밍** — 가장 자주 보게 되는 영역. 이름이 잘못되면 코드를 읽는 비용이 증가한다. 다만 설계가 바뀌면 이름도 따라 바뀌므로 설계 다음이다.
3. **테스트 커버리지와 품질** — 테스트가 없는 코드는 리팩터링이 불가능하다. 실패 케이스 누락은 프로덕션 버그로 직결된다.
4. **코드 품질 기본** — 매직 넘버, final, 포매팅, 접근 제한자. 자동화할 수 있는 부분이지만 놓치면 가독성이 떨어진다.
5. **성능** — 가장 관대한 영역. 명백한 낭비(매번 객체 생성, N+1 쿼리)만 짚고, 미시적 최적화는 넘어간다.

## 리뷰 톤 가이드

내가 지향하는 리뷰 커뮤니케이션 방식:

### 본받을 방식
- **질문형 피드백**: 지시보다는 질문으로 생각을 유도한다. "왜 이렇게 하셨나요?"는 상대방이 스스로 근거를 돌아보게 만든다.
  > "이런식으로 관리해주신 이유가 궁금해요!" — PR #102 (조앤)
- **근거 자료 첨부**: 피드백에 참고 링크나 코드 예시를 함께 제공하면 설득력이 높아진다.
  > "원시 타입을 포장해보면 어떨까요?" + 테코블 링크 첨부 — PR #661 (로키)
- **칭찬 먼저, 개선점 뒤에**: 잘한 부분을 먼저 인정하고 개선점을 제안하면 수용도가 높아진다.
  > "적절한 날짜 설정 아주 좋습니다." 후 개선점 제안 — PR #78 (앤지)
- **트레이드오프 토론 유도**: 단순히 "이렇게 바꿔라"가 아니라, 다른 방식을 직접 구현해보도록 권유한다.
  > "상상만으로 이 방식이 맞아 아니야를 판단하기보다는 직접 구현해보고 비교해보시면 좋을 것 같네요." — PR #808 (범블비)
- **대안 제시 + 선택권 부여**: 반드시 수정할 것과 고민해볼 것을 구분한다.
  > "너무 변경이 많을 것 같다면 꼭 하지 않으셔도 됩니당" — PR #102 (조앤)

### 피드백 시 주의할 점
- 약점을 지적하는 것이 아니라, "이 경험을 통해 배울 수 있는 것"으로 프레이밍한다
- 포매팅 같은 반복 지적은 한 번에 모아서, 핵심 설계 피드백에 집중한다
- "정답"을 제시하기보다 "왜?"를 묻고 스스로 답을 찾도록 유도한다
- 코드의 의도를 먼저 이해하려고 노력한 뒤 피드백한다

- **네이밍의 중요성**: 시간이 지날수록 더 확신. 초기에는 기본적인 네이밍 컨벤션 피드백을 받았지만, 후기에는 "객체가 사용 맥락을 모르는 이름"까지 고민하게 됨.
  > 초기: "bonus 보다는 bonusNumber" — PR #521 → 후기: "Hand가 '공개'라는 맥락을 알 필요가 없다" — PR #853
- **책임 분리**: 초기에는 DTO 변환 위치 정도였으나, 후기에는 상속 계층의 분류 기준 일관성, 전략 패턴까지 깊어짐.
  > 초기: "Dto가 도메인 객체에 의존해서 DTO를 생성하도록" — PR #521 → 후기: "하위 객체들이 일관된 기준이 아닌 서로 다른 기준으로 나뉜 것" — PR #102

### 완화된 기준
- **상속 vs 조합**: 초기(blackjack)에는 "상속은 무조건 지양"이라는 인상을 받았지만, 후기(janggi)에서 조앤의 피드백으로 "결합도를 높여야 하는 설계에서는 상속이 정답일 수도 있다"는 유연한 시각을 얻음.
  > 초기: "일반 클래스의 상속은 지양" — PR #808 → 후기: "결합도를 높여야 하는 설계에 있어서는 상속이 정답이 될 수도 있는거죠!" — PR #102

### 성장 요약
리뷰이로서 4명의 리뷰어에게 피드백을 받으며, 초기의 "기본적인 코드 품질 개선"(네이밍, final, 매직넘버) 단계에서 "설계 트레이드오프를 스스로 분석하고 근거를 대는" 단계까지 성장했다. 특히 상속 vs 조합, 인터페이스 도입의 목적, 도메인 모델링에 대한 이해가 깊어졌으며, 후반부에는 리뷰어에게 먼저 설계 질문("이런 상황에서는 어떤 설계를 적용하는 것이 더 좋을까요?")을 던지는 능동적인 자세를 보였다.
