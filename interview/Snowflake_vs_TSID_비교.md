### Question
Snowflake 대신 TSID를 선택한 결정적인 이유는 무엇이었나요?

### Answer
스노우플레이크는 TSID와 차이를 말해보자면 데이터 센터나 서버 노드 번호를 필수적으로 할당해야 되는 비트가 고정적으로 있었습니다. 그렇지만 저희는 단일 서버로 구성되어 있고 이런 부분이 불필요한 부분인데 꼭 구성해야 하는 부분, 또 이 부분 설정이 불편한 점을 고려해서 TSID에선 이 부분을 내부적으로 알아서 처리하고 꼭 설정하지 않아도 된다는 강점이 있어서 TSID를 활용하기로 했습니다.

### Feedback
Snowflake의 구조적 특성(데이터센터/워커 ID 필수 할당)과 현재 인프라 상황(단일 서버)을 연결하여 판단한 점이 좋습니다. 다만 두 기술의 비트 구조를 좀 더 구체적으로 설명하고, 향후 서버 확장 시 TSID가 어떻게 대응 가능한지까지 언급하면 더 완성도 높은 답변이 됩니다.

### Recommendation
"Snowflake는 64비트 중 타임스탬프 41비트, 데이터센터 ID 5비트, 워커 ID 5비트, 시퀀스 12비트로 구성됩니다. 분산 환경에서 충돌을 방지하려면 데이터센터와 워커 ID를 반드시 설정해야 하고, 이 설정을 잘못하면 ID 충돌이 발생할 수 있습니다.

반면 TSID는 64비트 중 타임스탬프 42비트와 랜덤/카운터 22비트로 구성되어, 별도의 노드 설정 없이도 사용 가능합니다. 저희는 현재 단일 서버 환경이라 Snowflake의 분산 설정이 오버엔지니어링이었고, TSID는 설정 복잡도 없이 바로 적용할 수 있었습니다.

또한 향후 서버를 확장하더라도 TSID는 node ID를 선택적으로 설정할 수 있어서 확장성 측면에서도 문제가 없다고 판단했습니다."
