### Question
Index Skip Scan을 사용할 때랑 세컨더리 인덱스로 조회할 때 커버링 인덱스가 적용되는 거랑 성능에 차이가 있을까요?

### Answer
커버링 인덱스 같은 경우에는 더 이상의 클러스터드 인덱스 조회를 하지 않아도 된다는 강점이 있는 것으로 알고 있고, Index Skip Scan 같은 경우에는 선두 컬럼 카디널리티가 낮아서 선두 컬럼을 제외하고도 인덱스를 사용할 수 있다는 점이 있어서 일대일 상대적인 비교가 조금 어려울 수도 있을 것 같습니다.

(추가 설명 후) 저 같은 경우에는 Index Skip Scan을 겪었던 경우가 세컨더리 인덱스가 아니라 클러스터드 인덱스였었기 때문에, 아무래도 결과적으로 데이터를 탐색하기 위해 디스크나 버퍼풀에 담겨져 있는 데이터를 가져왔을 것이고, 세컨더리 인덱스를 활용한 커버링 인덱스에서 조회를 했을 경우에는 더 이상의 디스크 I/O 없이 세컨더리 인덱스 리프에 있는 인덱스 값만을 보고 조회를 완료했을 것 같습니다.

### Feedback
처음에 질문 의도를 정확히 파악하기 어려웠지만, 힌트를 받은 후 디스크 I/O 관점에서 두 기법의 차이를 명확하게 설명해주셨습니다. 특히 "클러스터드 인덱스에서 Skip Scan이 발생했다"는 본인 경험과 연결하여, 커버링 인덱스는 추가 디스크 I/O가 없다는 핵심을 정확히 짚어주셨습니다. 면접에서 질문 의도를 되물어보는 것도 좋은 커뮤니케이션입니다.

### Recommendation
"두 기법은 해결하는 문제가 다르지만, 디스크 I/O 관점에서 비교해보겠습니다.

커버링 인덱스는 쿼리에 필요한 모든 컬럼이 세컨더리 인덱스에 포함되어 있어서, 클러스터드 인덱스를 추가로 조회할 필요가 없습니다. 세컨더리 인덱스의 리프 노드만 읽으면 되므로 디스크 I/O가 최소화됩니다.

Index Skip Scan은 복합 인덱스에서 선두 컬럼 없이도 인덱스를 활용할 수 있게 해주는 기법입니다. 하지만 Skip Scan 자체는 커버링 인덱스 여부와 별개입니다.

제 경험에서는 클러스터드 인덱스(PK)에서 Skip Scan이 발생했습니다. 클러스터드 인덱스의 리프 노드에는 실제 행 데이터가 저장되어 있으므로, Skip Scan으로 탐색하더라도 결국 데이터 페이지에 접근해야 했습니다.

만약 세컨더리 인덱스에서 Skip Scan이 발생하고, 동시에 커버링 인덱스 조건도 만족한다면, 클러스터드 인덱스 조회 없이 세컨더리 인덱스만으로 결과를 반환할 수 있어 디스크 I/O 측면에서 가장 효율적일 것입니다.

정리하면, Skip Scan은 '인덱스 접근 방식', 커버링 인덱스는 '추가 조회 회피 여부'로 서로 다른 축의 개념이며, 두 가지가 함께 적용될 때 최적의 성능을 낼 수 있습니다."
