### Question
클러스터드 인덱스와 세컨더리 인덱스에서 이 영향이 다르게 나타날까요?

### Answer
다르게 나타난다기보다는 조금 더 적은 영향으로 나타날 것 같습니다. 아무래도 클러스터드 인덱스 같은 경우에는 리프 노드에 실제 데이터들이 담겨 있기 때문에 디스크 I/O가 훨씬 더 많이 일어날 것 같고요. 클러스터드 인덱스가 아니었다면 아무래도 리프 노드도 PK로만 구성되어 있을 거고, 그런 상황임에도 인덱스의 정렬된 트리 구조는 동일하기 때문에 아마도 성능 차이가 조금 덜 나타나는 것으로 나타났을 것 같습니다.

### Feedback
클러스터드 인덱스의 리프 노드에 실제 데이터가 저장된다는 점, 그리고 세컨더리 인덱스의 리프 노드에는 PK만 저장된다는 점을 정확히 이해하고 계십니다. "데이터 크기 차이로 인해 I/O 영향이 다르다"는 핵심 논리도 맞습니다. 다만, 세컨더리 인덱스가 결국 클러스터드 인덱스를 참조해야 한다는 점과 페이지 크기 관점에서의 설명을 추가하면 더 완성도 높은 답변이 됩니다.

### Recommendation
"네, 영향의 정도가 다르게 나타납니다.

클러스터드 인덱스의 경우 리프 노드에 실제 행 데이터 전체가 저장됩니다. 따라서 한 페이지(기본 16KB)에 담을 수 있는 레코드 수가 적고, 랜덤 삽입 시 페이지 스플릿이 발생하면 이동해야 할 데이터 양이 큽니다. UUID처럼 랜덤 PK를 사용하면 이 오버헤드가 심각해집니다.

반면 세컨더리 인덱스의 리프 노드에는 인덱스 컬럼 값과 PK만 저장됩니다. 데이터 크기가 작아 한 페이지에 더 많은 엔트리가 들어가므로, 랜덤 삽입의 영향이 상대적으로 적습니다.

하지만 중요한 점은, 세컨더리 인덱스를 통해 데이터를 조회할 때는 결국 PK로 클러스터드 인덱스를 다시 조회해야 합니다. 이때 PK가 랜덤 값이면 클러스터드 인덱스에서 랜덤 I/O가 발생합니다. 결국 PK의 정렬 여부는 세컨더리 인덱스 조회 성능에도 간접적으로 영향을 미칩니다.

정리하면, 클러스터드 인덱스에서 영향이 가장 크고, 세컨더리 인덱스는 삽입 시에는 영향이 적지만 조회 시에는 클러스터드 인덱스를 거치므로 여전히 영향을 받습니다."
